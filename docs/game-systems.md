# Game systems

This doc is the home for Helio's actual game-mechanics, save-state, and roadmap context as it grows — what's a real mechanic versus presentation scaffolding, how the three kinds of state are kept apart, and where things are headed. The detailed design docs live in the gitignored `plans/` directory (see "Design docs" below).

Almost everything in this repo so far is the **map + system viewer** — the explorable shell. Actual game mechanics are only just starting; the status below tracks what's real.

**Where we are**

- **Explorable shell — mature.** The galaxy view (`StarmapScene`) and the close-up system view (`SystemScene`) are built out: ~1500 procgen stars (grouped into ~1200 systems), 8000+ bodies, full disc / atmosphere rendering, hover inspection. The shell's rendering is content + presentation; game-state layers (like the economy's cargo-traffic overlay) ride on top of it.
- **Game shell — the persistent sidebar.** A right-edge sidebar (AppController-owned, present in both views, surviving the view switch) carries the **turn controls** at the top — a turn counter + Next Turn, the first turn-state, persisted in the save (Next Turn now steps the live economy sim, then the sidebar re-reads its balances) — over a per-view contextual region: the galaxy's civilization summary + selected-system details + View System/Focus, and the system view's selected-body facilities. The 3D content insets to its left rather than hiding behind it.
- **Economy sim — wired and live.** A deterministic single-tier logistics core lives under `sim/` (read [the economy sim doc](../sim/README.md)). The app now drives it through `src/facilities/economy-bridge.ts`: each Next Turn steps it; placed facilities project into it (a **cluster is one economic node** — a system with a shared pool of bodies, so intra-cluster transport is free *and instant* (a same-system move is delivered the same turn, 0 turns of transit, never shown in flight) and only crossing between clusters costs jump range and time); and the economy reads back both as per-body / per-system surplus-deficit chips in the sidebar and as a forward-looking cargo-traffic overlay on the system diagram. Its full state persists across reloads. The viz is **predictive**: a throwaway clone of the world stepped one turn ahead drives the ship overlay and `++ inbound next turn` sidebar hints, so a placed provider's cargo and a relieved colony show *before* you commit the turn — the same `engine.step()` run early on a copy that is never persisted.
- **Facility construction.** The first real *game* mechanic and the first real save: select a planet / moon / belt in the system view and place a **colony**, **mining base**, or **farm** on it (Add pills in the right sidebar), persisted across reloads. Every fact about a facility type now lives in one registry — [`src/facilities/`](../src/facilities/README.md): its save-key, label, Add-button order, body-eligibility, build cap, and its economic projection. Eligibility is structural (any solid body; one of each type per body — no body-physics gate), and the economic projection (`facility → PlanetSpec`) is now **live**: a placed facility projects a **flat** contribution into the running sim (farms make food, mines make minerals, colonies eat both), accumulates stock turn over turn, and shows its per-resource balance in the sidebar (cost + ownership are still unbuilt).
- **Ship building.** The second real *game* mechanic. A **shipyard** is one more facility (the same Add-pill machinery, plus an `enablesShipbuilding` capability flag) — placing one surfaces a **Build ship** action on that body in the sidebar. A build takes a number of turns; cost is **time only** for now (a minerals-spend seam is reserved, not built), progress shows in the selection sidebar, and the build stepper runs as a turn phase in `AppController.nextTurn`. A completed ship joins the system's **fleet** — a single formation rendered in the open field below the planet row. Ship classes live in a neutral [`src/ships/`](../src/ships/README.md) registry. A fleet ship is now **selectable** — clicking it opens a read-only ship card in the sidebar (name / class / status) — but has no **actions** yet (no selection rim, no movement); those plus health/energy land with movement and the encounter-combat layer. Design: `plans/4x-ship-building-system.md`.

**The save-state layering** (the one bone set carefully) — three kinds of state, three homes, kept apart on purpose:

- **Catalog / skeleton** — stars + bodies, the gitignored `catalog.generated.json`. Static *content*, regenerated from CSV; never a save.
- **Game state** — `src/game-state.ts`, a versioned `localStorage` JSON blob (`helio.game`) mirroring `settings.ts`. Stores player *intent*: the current **turn** number (1-based; added without a version bump — reads merge over defaults) plus placed facilities, keyed by the **stable** `Body.id` (resolved back to an index via `indexOfBodyId` in [the catalog data doc](../src/data/README.md)), with **skip-on-missing** load — a facility whose body a catalog rebuild dropped (a CSV id change or `PROCGEN_VERSION` bump) is discarded with a DEV warning, never fatal. Built **ships** share this blob and its `seq` id counter (the `'s'`/`'f'` prefix disambiguates), but key to their **system** — a stable cluster-primary star slug, never a planet — because a ship is a peer of planets, not an appendage, and must outlive its birth shipyard's body; the same skip-on-missing discipline drops a ship whose system (or, while still building, whose shipyard body) the catalog no longer carries.
- **Sim state** — the economy sim's own bit-stable binary save, now live: base64 under the separate `localStorage` key `helio.sim`, `configHash`-guarded (a catalog/scale/resource change cold-starts instead of mis-loading). Deterministic, integer.

Facilities are deliberately **not** in the sim's save: the sim has no concept of bodies (only abstract economic nodes), and a facility stores no economic behavior. The projection that derives that behavior — `Body + Facility[] → PlanetSpec`, summing each body's facilities into one node — lives in `src/facilities/project.ts` + `economy-bridge.ts`, so the `helio.game` save shape stays `{ bodyId, type }`. The derived economic *state* (stock, in-flight cargo, smoothing) **is** persisted — in the sim's `helio.sim` binary save, not recomputed-and-discarded — so a reload resumes the same economy; a mid-game build reconciles by `Body.id` rather than re-projecting, so it never zeroes accumulated stock. `STORAGE_KEY` is the single seam for the planned multi-slot saves.

**Reasonable next steps** (rough order, none committed):

- A selection rim visually distinct from the hover rim (today they share one).
- More facility types beyond `colony` / `mining-base` — now one `FacilityDef` object each; the sidebar shows one Add pill per type the selected body can host, which will want a proper picker once the list grows past a handful.
- Deepen the now-live economy: build costs + build time, warehouse/depot storage caps (the `storageCeiling` lever, now inert — producers are demand-pull faucets, so it's reserved for a finite larder/depot), trade-hub depot nodes, and a galaxy-view edge-flow overlay (the `digest.edgeFlows` read surface is ready; the 3D line layer isn't built).
- Build out ships past v1: a minerals cost (the reserved spend seam), ship **actions** (a selection rim + verbs to pair with the now-selectable read-only ship card), movement between systems, and the encounter-combat layer that consumes the fleet (`plans/4x-encounter-combat-system.md`) — at which point ships gain health/energy as peers of planets.
- Planet ownership / players.
- Multi-slot saves with a new-game / load-game splash (the `STORAGE_KEY` seam).
