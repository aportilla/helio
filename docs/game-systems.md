# Game systems

This doc is the home for Helio's actual game-mechanics, save-state, and roadmap context as it grows — what's a real mechanic versus presentation scaffolding, how the three kinds of state are kept apart, and where things are headed. The detailed design docs live in the gitignored `plans/` directory (see "Design docs" below).

Almost everything in this repo so far is the **map + system viewer** — the explorable shell. Actual game mechanics are only just starting; the status below tracks what's real.

**Where we are**

- **Explorable shell — mature.** The galaxy view (`StarmapScene`) and the close-up system view (`SystemScene`) are built out: ~1500 procgen stars (grouped into ~1200 systems), 8000+ bodies, full disc / atmosphere rendering, hover inspection. The shell's rendering is content + presentation; game-state layers (like the economy's cargo-traffic overlay) ride on top of it.
- **Game shell — the persistent sidebar.** A right-edge sidebar (AppController-owned, present in both views, surviving the view switch) is a **three-band frame**: a fixed **turn header** (a turn counter + Next Turn, the first turn-state, persisted in the save — Next Turn steps the live economy sim, then the sidebar re-reads its balances — plus the settings glyph), a **scrolling body** (a `ScrollView` wrapping the per-view contextual region: in the galaxy, a top-level game-views menu when nothing is selected or the selected system's ship list; in a system, the selected body's facilities / ship card), and a fixed **footer** of contextual nav actions (galaxy: pan/zoom, or View System / Deselect for a selected system). The 3D content insets to its left rather than hiding behind it.
- **Economy sim — wired and live.** A deterministic single-tier logistics core lives under `sim/` (read [the economy sim doc](../sim/README.md)). The app now drives it through `src/facilities/economy-bridge.ts`: each Next Turn steps it; placed facilities project into it (a **cluster is one economic node** — a system with a shared pool of bodies, so intra-cluster transport is free *and instant* (a same-system move is delivered the same turn, 0 turns of transit, never shown in flight) and only crossing between clusters costs jump range and time); and the economy reads back both as per-body surplus-deficit chips in the system-view sidebar and as a forward-looking cargo-traffic overlay on the system diagram. Its full state persists across reloads. The viz is **predictive**: a throwaway clone of the world stepped one turn ahead drives the ship overlay and `++ inbound next turn` sidebar hints, so a placed provider's cargo and a relieved colony show *before* you commit the turn — the same `engine.step()` run early on a copy that is never persisted.
- **Facility construction.** The first real *game* mechanic and the first real save: select a planet / moon / belt in the system view and place a **colony**, **mining base**, or **farm** on it (Add pills in the right sidebar), persisted across reloads. Every fact about a facility type now lives in one registry — [`src/facilities/`](../src/facilities/README.md): its save-key, label, Add-button order, body-eligibility, build cap, and its economic projection. Eligibility is structural (any solid body; one of each type per body — no body-physics gate), and the economic projection (`facility → PlanetSpec`) is now **live**: a placed facility projects a **flat** contribution into the running sim (farms make food, mines make minerals, colonies eat both), accumulates stock turn over turn, and shows its per-resource balance in the sidebar (cost + ownership are still unbuilt).
- **Ship building.** The second real *game* mechanic. A **shipyard** is one more facility (the same Add-pill machinery, plus an `enablesShipbuilding` capability flag) — placing one surfaces a **Build ship** action on that body in the sidebar. A build takes a number of turns; cost is **time only** for now (a minerals-spend seam is reserved, not built), progress shows in the selection sidebar, and the build stepper runs as a turn phase in `AppController.nextTurn`. A completed ship joins the system's **fleet** in the open field below the planet row, where ready ships render in two opposing formations — the player's side at left facing right, opponents at right facing left, a stand-off staged across the field. A ship has **no class** — it carries its own ordered list of **modules** (`components`, persisted in the save), and everything about it (actions, combat installs, energyMax, build time, and its on-screen look — a segmented hull of kind-colored rects framed in its faction color, drawn rear→nose) derives from those modules; the neutral [`src/ships/`](../src/ships/README.md) registry defines the modules. A fleet ship is **selectable** — clicking it opens a read-only ship card in the sidebar (name / faction / status) **and pops an anchored action menu** — selecting a controlled **actor** (a fleet ship OR a facility-bearing planet/moon/belt) drills category → command → target → confirm (mouse + keyboard parity), offensive verbs targeting enemy ships **and** enemy bodies alike: the **system action menu**, the general "how you act on the world" grammar of which combat is one consumer. Its committed actions dispatch by kind: the `'immediate'` verbs still route to no-op stubs, but the `'encounter'` hand-off is **live** — it enters the combat encounter mode (first-playable, E1–E4; see [src/encounter/README.md](../src/encounter/README.md)), where ships gain pool-stack health + an energy gate as combatants. Ship **movement** between systems has now shipped (the next paragraph). Design: [`src/actions/`](../src/actions/README.md) + `plans/4x-ship-building-system.md`.

  **Ship movement (warp).** A ready ship warps between systems, single-hop, within its drive's range. Navigation is a **galaxy-view modality**, not a system-view command: selecting a cluster lists its ready player ships in the sidebar (the **SHIPS** block), and clicking one enters a transient **departure mode** on the galaxy view — a range ring at the drive's reach, in-range systems lit and the rest dimmed, lock-then-confirm a destination. Confirming flips the ship to `'transiting'` (its `systemId` stays the origin until arrival) and it drops out of the SHIPS list, counting down to an `arrivesOnTurn` resolved by a `stepShipTransits` turn phase; a dotted transit line + a faction-coloured progress head track it on the galaxy map, and the system sidebar's TRANSITS block lists outbound/inbound rows. **Warp range equals the economy's trade reach** (one adjacency map — fleets go exactly where trade can, pinned by test) and movement is **energy-inert** (a ship arrives fully charged). The substrate is **faction-agnostic** — an `ArrivalEvent` stream + a turn-stepped order API the eventual rival mover reuses with no UI. Design absorbed here + into [`src/scene/`](../src/scene/README.md) / [`src/actions/`](../src/actions/README.md). When a ship **arrives**, its own muster **sprite flies in** onto the berth reserved while it was inbound — the **first time you open its destination system that turn**, not only if you happened to be watching at Next Turn (`AppController` holds a per-system pending-arrival set the system view consumes on first display; it never replays on a re-view). No separate FX layer — it's the real `FleetLayer` sprite translating. The formation carries stable **berths**: a departing ship leaves a **gap** (no back-fill) and an inbound ship **reserves the spot** it warps into, so a warp never reflows the other ships. `?demo-warp` drives the fly-in/out animation.

  Ships now carry an **owner** — the first sliver of the ownership model. A faction is a neutral side in its own registry, [`src/factions/`](../src/factions/README.md) (a twin of `src/ships/`), modelled deliberately narrow: **species-agnostic**, with no "is-human" flag — *which* faction the local player commands is a separate pointer (`CONTROLLED_FACTION_ID`), never a property of the faction record. Each ship stores a `factionId`; the fleet sprite and the ship card tint by it, so the player's ships and any opposing ships in the same system read apart at a glance. To exercise this (and the encounter-combat layer) ahead of real movement/diplomacy, **DEV-only** sidebar actions drop a ready opponent ship into the focused system and claim the selected body for an opponent (a **+ opponent colony** pill that places an opponent colony — a facility + an ownership flip, the facility being what claims it) — debug scaffolding kept out of production builds and out of the durable test surface, not player-facing mechanics.

**The save-state layering** (the one bone set carefully) — three kinds of state, three homes, kept apart on purpose:

- **Catalog / skeleton** — stars + bodies, the gitignored `catalog.generated.json`. Static *content*, regenerated from CSV; never a save.
- **Game state** — `src/game-state.ts`, a versioned `localStorage` JSON blob (`helio.game`) mirroring `settings.ts`. Stores player *intent*: the current **turn** number (1-based; added without a version bump — reads merge over defaults) plus placed facilities, keyed by the **stable** `Body.id` (resolved back to an index via `indexOfBodyId` in [the catalog data doc](../src/data/README.md)), with **skip-on-missing** load — a facility whose body a catalog rebuild dropped (a CSV id change or `PROCGEN_VERSION` bump) is discarded with a DEV warning, never fatal. Built **ships** share this blob and its `seq` id counter (the `'s'`/`'f'` prefix disambiguates), but key to their **system** — a stable cluster-primary star slug, never a planet — because a ship is a peer of planets, not an appendage, and must outlive its birth shipyard's body; the same skip-on-missing discipline drops a ship whose system (or, while still building, whose shipyard body) the catalog no longer carries. A ship also stores its **owner** (`factionId`) — validate-and-merged to the controlled faction for any ship saved before ownership existed, so adding the field can't invalidate an old save — and its build-only fields (shipyard body, completion turn) only while `'building'`: a `'ready'` ship that never went through the build flow carries neither.
- **Sim state** — the economy sim's own bit-stable binary save, now live: base64 under the separate `localStorage` key `helio.sim`, `configHash`-guarded (a catalog/scale/resource change cold-starts instead of mis-loading). Deterministic, integer.

Facilities are deliberately **not** in the sim's save: the sim has no concept of bodies (only abstract economic nodes), and a facility stores no economic behavior. The projection that derives that behavior — `Body + Facility[] → PlanetSpec`, summing each body's facilities into one node — lives in `src/facilities/project.ts` + `economy-bridge.ts`, so the `helio.game` save shape stays `{ bodyId, type }`. The derived economic *state* (stock, in-flight cargo, smoothing) **is** persisted — in the sim's `helio.sim` binary save, not recomputed-and-discarded — so a reload resumes the same economy; a mid-game build reconciles by `Body.id` rather than re-projecting, so it never zeroes accumulated stock. `STORAGE_KEY` is the single seam for the planned multi-slot saves.

**Reasonable next steps** (rough order, none committed):

- A selection rim visually distinct from the hover rim (today they share one).
- More facility types beyond `colony` / `mining-base` — now one `FacilityDef` object each; the sidebar shows one Add pill per type the selected body can host, which will want a proper picker once the list grows past a handful.
- Deepen the now-live economy: build costs + build time, warehouse/depot storage caps (the `storageCeiling` lever, now inert — producers are demand-pull faucets, so it's reserved for a finite larder/depot), trade-hub depot nodes, and a galaxy-view edge-flow overlay (the `digest.edgeFlows` read surface is ready; the 3D line layer isn't built).
- Build out ships past v1: a minerals cost (the reserved spend seam) and richer ship **actions**. Ship **movement (warp)** and the **actions** menu have shipped (above). The **encounter-combat layer** that consumes the fleet is now **first-playable** (E1–E4 + the effect substrate — ships gain pool-stack HP + an energy gate in combat — plus the **event-animation lifecycle**: an action animates as a beat of per-weapon-coloured barrage bolts source→target with an impact flash + damage-number pop, EV §14; `plans/4x-encounter-combat-system.md`). The reducer's last hard-coded attack branch has been retired — damage is now a declared effect — and on that a first slice of real **dynamic mechanics** is in: weapons with asymmetric effectiveness (a laser strips shields, a cannon craters hull) and an always-on shield that regenerates, costs energy, and **fritzes out** when stripped (see [src/encounter/README.md](../src/encounter/README.md)). What remains there is the real damage formula + PRNG (P-Experiment), body combatants (E5), richer opponent AI, and the event-animation tail (heal/shield event beats).
- Planet ownership / players — building out the faction model (`src/factions/`). A first sliver of **body** ownership has shipped: a `BodyOwnership` overlay + `ownerFactionId` (a body with no record defaults to the controlled faction), with the economy fan-in ownership-gated so an enemy-held body can't feed the player, and a DEV claim pill. What remains: real faction names/identities, capture/transfer write-paths, and a controlled-faction pointer that can change hands.
- Multi-slot saves with a new-game / load-game splash (the `STORAGE_KEY` seam).
