# src/scene — 3D scene architecture + rendering

This is the Three.js scene layer — the galaxy view and the system view, the `AppController`/scene split that swaps the active scene over a shared `WebGLRenderer`, the camera + input model, and ALL of the rendering/shader/pixel-art detail. The modules here know **nothing about the DOM** beyond the `HTMLCanvasElement` they render into and `window` for size/input listeners (the precise, narrow allowance is in the DOM-boundary note below). The pixel-crisp aesthetic and the ColorManagement-OFF constraints documented here are project-wide commitments, not local choices — treat them as constraints the rest of the codebase leans on, not defaults to grow past.

## Bootstrap / scene split

`main.ts` is minimal — it imports the global stylesheet, parses the bundled BDF fonts, creates the canvas, instantiates an `AppController`, and dismisses the boot splash (held 350 ms, faded 600 ms). The controller owns the shared `WebGLRenderer` and decides which view-mode scene's `tick()` loop is currently driving the canvas. Three peer scenes share the renderer: `StarmapScene` (galaxy view, the default), `SystemScene` (close-up of one cluster, lazily constructed on entry, disposed on exit), and `TestScene` (a static planet-render test grid behind the galaxy HUD's top-left trigger, also lazy/disposed — a dev tool for eyeballing a disc-tuning edit across a frozen-surface aging sweep — bulkWaterFraction (rocky snowball ↔ ice-shell) × age, at a fixed seed — at once). Only one is running at a time.

The boot splash itself (cyan dot + two pinging range rings) is **inlined in `index.html`** — markup in `<body>`, critical CSS in a `<head>` `<style>` block — so it paints on the first frame, before the JS bundle is fetched and parsed. The splash isn't waiting on `main.ts`; `main.ts` is just responsible for removing it once the scene is up.

There's no UI plumbing between the bootstrap and the scenes: each scene owns its own HUD orchestrator (`MapHud` for galaxy view, `SystemHud` for system view), each with its own ortho pass at 1 unit = 1 buffer pixel. HUD widgets are built on `Widget` (Mesh + PlaneGeometry + MeshBasicMaterial + optional CanvasTexture) so HUD geometry shares the rest of the scene's pixel grid.

Each HUD captures pointer events first (in the scene's `onPointerDown` / `onPointerMove` via `clientToHud()`), and the persistent sidebar is consulted before the HUD, so clicking a button, a panel row, or the sidebar never starts a pan/orbit and hovering swaps the cursor to `pointer`. `MapHud` exposes `onToggle`, `onAction`, `onViewTest`, and `onSettingsChanged` callbacks (the selection-driven actions — View System, Focus, deselect — moved to the sidebar's `GalaxyContext`); `SystemHud` exposes `onBack`. `src/settings.ts` is the single source of truth for persisted user preferences — `singleTouchAction`, the display toggles (`showLabels`, `showDroplines`), and `resolutionPreference`. The blob is versioned; old saves are merged over fresh defaults on read so adding new fields can't invalidate them. `MapHud` seeds its in-panel toggle state from `getSettings()` at construction and writes through `setSetting()` on each flip; the scene seeds the renderer-side state (`Labels`, `Droplines`) from the same source so the HUD checkbox and the rendered geometry agree on first paint. For touch input and resolution, the scene reads `getSettings()` at gesture/resize time (pull-on-read), so a flipped preference takes effect on the next pointer event or resize with no callback plumbing — `onSettingsChanged` triggers a `resize()` so the resolution radio applies immediately rather than waiting for a window event. The `spin` toggle is intentionally NOT persisted — it's a session fidget, not a preference that should survive a refresh.

Each HUD also exposes `hitTest(bufX, bufY)` returning one of `'interactive' | 'opaque' | 'transparent'` (see `src/ui/hit-test.ts`). The scene queries it once per `pointermove` and only sets `pointer.has = true` when the result is `'transparent'`, so stars behind a button, panel, or the sidebar don't light up hover labels through the chrome above them. `'opaque'` covers visually-solid surfaces that aren't clickable (panel background, sidebar surface) — they block scene picks and absorb clicks (so a mousedown there doesn't start an orbit drag) without changing the cursor. `'interactive'` adds the cursor swap and dispatches the click. This three-way model is the seed of a layered input router: each HUD's `hitTest` will become an `InputLayer` when modals / tooltips / context menus arrive.

The `scene/` modules touch the DOM only through the `HTMLCanvasElement` they render into and `window` (size/input listeners) — plus detached offscreen `<canvas>` elements created purely as 2D drawing surfaces for textures (never attached to the document, never queried). No DOM queries, nothing reaching into the page: cursor state is set inline on the render canvas (`canvas.style.cursor`), not on `document.body`. Don't widen this — route data through callbacks or new methods on the scene. There's a separate, deliberate dependency from `scene/` ONTO `ui/`: the scene **consumes** ui — it builds the HUD orchestrators (`map-hud`, `system-hud`), renders the whole persistent `sidebar`, queries the `hit-test` contract, and reads `ui/theme`'s spacing tokens (`sizes.edgePad`) so its margins match the rest of the chrome. The invariant worth naming is the **direction**: ui never imports scene (verified acyclic). The shared `diagram-pick.ts` pick contract is hoisted to the src/ root precisely to keep it that way — both layers depend on it without ui having to reach back into scene. So they are not import-peers; scene depends on ui, never the reverse.

## Coordinate system

Galactic cartesian, units in light years:
- **+X** points toward the galactic centre (where the +X arrow points)
- **+Z** points toward the north galactic pole (the camera's up vector is fixed to `(0, 0, 1)`)
- The Sun sits at the origin

Star positions are derived at build time from each Wikipedia row's RA/Dec/distance via the standard ICRS → galactic rotation matrix (J2000), so they're as accurate as Wikipedia's parallax-derived inputs. Don't treat the catalog as scientifically authoritative anyway — Wikipedia itself rounds aggressively, and our radii are derived from class + mass (not measured) since the upstream table doesn't carry a radius column.

## Camera

`PerspectiveCamera`, FOV 45°. The camera orbits a 3D pivot point (`view.target`). Single-click on any star (no drag) snaps the pivot onto it; `view.target` lerps to the new star over ~400 ms (ease-in-out cubic) while yaw/pitch/distance stay frozen, so the camera glides over rather than swinging. WASD then translates the pivot in 3D (camera follows by the same vector, distance preserved) so the user can pan away from any star and orbit empty space — clicks set the pivot, keys drift it.

The orbit state lives in `view = { target, distance, yaw, pitch, spin }`. `distance` is the **camera-to-target orbit radius in light years** — closer = zoomed in. Wheel/pinch dolly the orbit in/out; bounds are `[ZOOM_MIN, ZOOM_MAX]` ly. Initial focus = the Sun; the HUD's "reset view" snaps focus, distance, yaw, and pitch back to their defaults instantly (a snap, not a glide — animating four axes at once looks jolty).

**Startup opening beat.** `StarmapScene.start()` arms a one-shot timer that auto-selects Sol shortly after start if the user hasn't picked anything by then. The selection fires the grid's staged ring-expand choreography (see "Sequential per-ring expand/collapse on selection change") on first paint, so the title screen settles into the catalog rather than appearing static. The timer is cleared in `stop()` and skipped if a selection already exists when it fires — clicking any star before the timer elapses cancels the auto-pick cleanly.

Drop-lines now converge toward a vanishing point; under perspective, that's the honest depth cue and we lean into it. The focused-star pivot does the orientation work that the parallel pins used to.

## System view

A close-up tactical view of one cluster lives in `SystemScene` (peer of `StarmapScene`, swapped in by `AppController`). Entry: clicking the **View System** pill in the galaxy sidebar, double-left-clicking a star, or pressing **Enter** with a cluster selected. Exit: the floating back button (top-left of the system view), or `Escape`.

`AppController` owns the shared `WebGLRenderer` and the persistent `StarmapScene` ROOT, with lazily-built overlays (system / test) layered on top through a small `OverlayStack` (`overlay-stack.ts`) — the depth-N generalization of a former single slot, so a modal can sit over the system view later (the encounter, eventually) without disturbing the round-trip. Only depth-1 is reachable today: the generic `enterOverlay` guards re-entry, so galaxy↔system/test behaves exactly as a single slot would. Galaxy view state — camera, selection, settings — lives on the `StarmapScene` instance, so the round-trip preserves it without any serialization: entering an overlay `stop()`s the root and constructs a fresh `SystemScene`; exiting disposes the overlay and `start()`s the screen beneath it (the root today). The galaxy scene's `tick()` is paused, not torn down, so resuming is instant and the camera comes back exactly where the user left it. `StarmapScene` also carries an idempotent `dispose()` (mirroring its siblings — releases the HUD, both bracket meshes, and the viewport's `RenderScaleObserver` window/`matchMedia` listeners), but it's reserved for app teardown via `AppController.stop()`, NOT the system round-trip, which deliberately only pauses with `stop()`/`start()`. Latent today since the starmap is a singleton; the contract is in place before a future screen swaps it out.

A live overlay can also **suspend the outer turn loop**: a `Screen.freezesTurn` marker short-circuits `AppController.nextTurn()`, and `Sidebar.setNextTurnEnabled(false)` greys + inerts the Next Turn pill — the two-point seam a modal pulls to run its own loop within a single galaxy turn (the encounter modal). Both are unused by the galaxy/system/test screens, which advance the turn normally; they default to "live."

`AppController` also owns the persistent **sidebar** (`src/ui/sidebar/`) — the right-edge chrome carrying the turn controls (a turn counter + Next Turn, backed by the game-state save) over a per-view contextual region. It's constructed once and *rendered, resized, and input-routed by whichever scene is active* (a final overlay pass; consulted before that scene's own HUD), so it survives the galaxy↔system round-trip with no rebuild — only its contextual region swaps (galaxy: civ summary + selected-system + View System/Focus; system: the selected body's facilities). The reserved strip width feeds `ViewportSizer`'s content rect so the 3D pass insets to its left rather than hiding behind it. `StarmapScene` reads the selected system's aggregate economy balance from the `EconomyBridge` (a type-only import) and pushes it into `GalaxyContext` on each selection change and after a turn, so the galaxy info-card's net balances track the live sim. See [the ui doc](../ui/README.md).

Entering the system view triggers a one-time GLSL compile/link of the three planet-material program variants (`disc`, `halo`, moon `all`). To keep that stall off the **View System** click frame, `AppController` warms them during galaxy-view idle — `renderer.compile()` on a throwaway scene (`warm-shaders.ts`) — and *retains* the warmed materials so their shared GL programs keep a non-zero ref-count across every `SystemScene` dispose, so re-entry never recompiles. Best-effort: a warm failure is swallowed and the system view simply pays the original compile.

`SystemHud` mirrors `MapHud`'s structure (own scene + ortho camera + composed widgets, `autoClear` off), but it's slim: an `IconButton` back-arrow in the top-left (the only click target) and a `BodyInfoCard` — a transient on-hover tooltip the scene shows whenever the pointer is over a disc in the diagram below (see "Hover info card" below). The system name and the **selected** body's facilities + *Add* controls live in the persistent sidebar's contextual region (`SystemContext`), not here — see "Body selection + facilities" below.

The scene's body is a flat 2D diagram, not a 3D space. `SystemDiagram` (in `src/scene/system-diagram/index.ts`) is a thin coordinator: it owns the Three.js `Scene` + `OrthographicCamera` (1 unit = 1 buffer pixel — the same convention as `Labels` and the HUD overlays), holds one layer object per render kind under `layers/`, and runs the layout/pick/hover passes through them in order. Layout has two zones stacked at fixed distances from the top edge, so the stars + bodies form a constant-height block hanging from the top and the empty space below (where the cargo-ship overlay flies — see **Cargo ships** below) grows or shrinks with the viewport:

- **Stars row.** Each cluster member is a `Mesh + PlaneGeometry` disc whose center sits *above* the buffer top by `STAR_OFFSCREEN_FRAC` of the disc radius, so the GPU clips the offscreen portion and the visible sliver reads as "huge body, mostly hidden up there". Mesh (not `Points`) is load-bearing: `gl.POINTS` discards any sprite whose vertex falls outside the clip volume, but the triangle path rasterizes fine with vertices outside the viewport. Disc sizes preserve the within-class ratios from the catalog's `pxSize` field, multiplied by `DISC_SCALE`. The row is pinned to the top-left (starts at `sizes.edgePad`, grows rightward) and laid out in catalog member order — `cluster.members[0]` is the primary (most massive), so it takes the leftmost slot with companions trailing right. The row narrows by scaling discs down (not by tightening the gap) once `STAR_HORIZ_GAP_FACTOR × largest-diameter` falls below `MIN_STAR_GAP`. Every star also gets a paired outer-halo mesh (a larger `Mesh + PlaneGeometry` sized to `STAR_HALO_RADIUS_FACTOR × disc radius`) rendered at `RENDER_ORDER_STAR_HALO` (negative) so it draws BEFORE every other body — later passes paint over it.
- **Body dome.** Planets sit on a peak-anchored arc below the stars. The peak is fixed at `PLANET_PEAK_FROM_TOP` env-px below the top edge; the edges drop by an amount that lerps from `DOME_PEAK_MIN_PX` to `DOME_PEAK_MAX_PX` as viewport *area* grows from `DOME_AREA_MIN` to `DOME_AREA_MAX` (env-px²). Area (rather than width or height alone) drives the lerp because the arc reads as "proportional to how much real estate you have." Planets **and belts** are placed in semi-major-axis order, innermost left to outermost right, with **even edge-to-edge gaps** — the free width is split into N+1 equal pieces (one between each adjacent pair plus a margin on each end), so a Mercury sitting next to a Jupiter (or a belt sitting between them) preserves the same visual gap as two same-sized neighbors. Dome Y is keyed to each slot's actual x rather than its ordinal index, so the peak stays at the row's geometric center regardless of within-row size variation. Moons straddle their parent's rim at angles seeded per planet and distributed by largest-gap-fill with geometric per-pair margins so adjacent moons land tangent rather than touching, then split into back/front pools so the upper half is occluded by the parent and the lower half overlaps downward.

- **Belts** consume a row slot the same way planets do (`BELT_SLOT_WIDTH` env-px) but render as a vertical scatter of angular polygon blobs centered on the slot's cx/cy and reaching `BELT_HEIGHT_FACTOR` × the largest planet disc above and below, capped at `BELT_HEIGHT_MAX_PX` so a row carrying a large gas giant doesn't stretch the column to the full viewport. Each chunk picks a silhouette from the belt's shape library (`POTATO_SHAPES` / `CRYSTAL_SHAPES`, selected by `shapesFor` — irregular convex polygons inscribed in the unit circle), a half-extent from `BELT_CHUNK_SIZES`, and a random rotation — seeded per-chunk off the belt's id so the silhouette stays stable across reloads. Two pieces of belt metadata shape the scatter. `largestBodyKm` (the size of the belt's largest parent body) sets chunk *size* — `beltChunkSizeScale` maps it log-scaled onto a per-belt multiplier (`BELT_CHUNK_SCALE_MIN`…`BELT_CHUNK_SCALE_MAX`) on the whole `BELT_CHUNK_SIZES` palette. `BELT_CHUNK_SIZES` is tuned for the large end, so the multiplier tops out at 1.0 (a Ceres/Pluto-class shepherded belt renders at the base palette) and scales *down* from there, so a trace dust cascade reads as fine gravel rather than boulders. The within-belt center-proximity size bias still applies on top. **Mass** then sets chunk *count*: a log-based mass count (between `BELT_CHUNKS_MIN` and `BELT_CHUNKS_MAX`) is divided by chunk area (the size multiplier squared) and capped at `BELT_CHUNKS_HARD_MAX`, so a belt of smaller bodies renders as *more* of them at the same belt mass — total painted area stays ≈ mass-driven regardless of body size, and a dust cascade reads as a dense fine swarm while a Ceres-class belt is a sparse handful of boulders. The whole belt pool is one indexed triangle Mesh drawn through `makeBlobMaterial` (flat-color triangle fill, per-vertex hover flag). Chunk **color** comes from the resource grid per chunk: each chunk's primary hue is one of the belt's two deposits (`dominantResources`), drawn weighted by abundance and painted in the muted single-resource mineralogy palette via `resourceMineralColor` (`ROCK_ARCHETYPE_SINGLE`) — so a metals+volatiles belt reads as a mix of grey rock and pale ice rather than one flat hue, and the proportion of each color tracks the deposit split. Each chunk also carries the belt's *other* deposit as a secondary hue plus a per-chunk random blend direction; the blob shader dithers a faint directional stipple (`OFFCOLOR_HINT` peak coverage, offset-Bayer keyed) of the secondary into the rock, so every chunk reads as its primary mineral tinged by the second deposit from one side rather than as a flat single color. The **shape** library is still belt-level: `shapesFor(bodyIcyness)` switches the whole belt from weathered boulders (`POTATO_SHAPES`) to faceted ice shards (`CRYSTAL_SHAPES`) once icyness crosses 0.5. Resource-less belts (defensive — every emit should set the grid) fall back to the rocky↔icy `beltRingColor` lerp. One chunk primitive — visual character emerges from the same resource grid that drives gameplay.

- **Rings** render as a tilted ellipse around the host planet — perspective-compressed by `RING_MINOR_OVER_MAJOR` (Saturn-flat) with the in-plane tilt derived from the **host planet's** `axialTiltDeg` via `bodyVisualTiltRad` (rings sit in the planet's equatorial plane by physics, so the ring and the banded atmosphere share one tilt). The mapping scales degrees by `AXIAL_TILT_VIZ_SCALE` and clamps to `±RING_TILT_DEG_MAX` so a Jupiter (3°) reads as nearly straight while a Uranus (97.8°) maxes out without breaking the back/front split. Bodies with null `axialTiltDeg` fall back to a seeded random off the body id so they don't all comb-align at 0. The ellipse's outer half-axis = `outerPlanetRadii × planetRadius`; inner = `innerPlanetRadii × planetRadius`. Each ring is two solid triangle-strip annulus halves (back at `RENDER_ORDER_BACK_RING`, front at `RENDER_ORDER_FRONT_RING`). Both halves run AFTER the planet pass so a translucent back-ring can blend over a left-neighbor's disc instead of against the cleared framebuffer (otherwise the disc would be depth-rejected behind a ring blended against background). Within R's own stack, R's back ring is hidden by R's disc via depth test — back ring at `z_R + Z_BACK_RING` fails `LessEqual` against the disc at `z_R`. Ring materials write depth even when translucent so a closer back-half blocks a farther planet's front-half from punching through. `RING_SEGMENTS` controls the silhouette resolution; each annulus vertex also carries `aRho` (0 inner edge → 1 outer edge) and `aAngle` (0..1 around the full ellipse, in phase across both halves) for the shader to read.

  `makeRingMaterial` renders the ring as a **stack of discrete concentric ringlines over a translucent floor** — no dither, no noise. The shader quantizes the radial span into `uRingCount` 1px-wide bands (the count is the ring's radial pixel width, set in `buildRing`) and samples everything at each band's center, so every fragment of a line shares one crisp constant tone — no gradient across a line. The ellipse's minor-axis compression naturally crowds many lines into a pixel toward the disc's top/bottom edges, the dense-edge read of a tilted Saturn. Each line's shade is a smooth density envelope (broad × fine seeded sine harmonics — the bright-zone/dim-zone grouping) times a per-line hash, with one Cassini-like division gap cut through the disc; the density drives both the line's brightness (`SHADE_DIM`…`SHADE_BRIGHT`) and its opacity (`ALPHA_MIN`…1 of the floor alpha), opacity kept high across the board so the disc stays present and the line-to-line read rides mostly on brightness. A per-ring seed off the body id varies the envelope frequencies / phases / gap position so two rings never comb-align. There's no discrete ice-vs-debris ring type — composition lives in the ring body's six-resource grid, and the renderer derives a 0..1 icyness scalar from it via `bodyIcyness()` (`resVolatiles` dominant → 1; rocky resources dominant → 0). That scalar lerps both the color (between `BELT_RING_COLOR_ROCKY` and `BELT_RING_COLOR_ICY`) and the floor opacity (between `RING_FLOOR_ALPHA_DUSTY` and `RING_FLOOR_ALPHA_ICY`): a Saturn-class ring (volatile-heavy) renders as a bright cyan-white band on a near-solid floor; a Uranus/Neptune-class ring (rocky-dominant) renders as a dark dusty band on a faint floor that reads through to the background; partials sit in the middle. The same data drives the resource panel on the hover info card, so visual character and gameplay attribute can't disagree.

  **Planet shadow.** `RingsLayer.layout()` resolves each ring's dominant star (brightest published light, tie-break nearest the host) and writes its screen-space direction plus the host center / normalized radii into the material. The shader reconstructs a faked-3D fragment position — screen x,y exact from the pixel, depth from the parametric angle scaled by the ring's viewing inclination (the un-drawn depth is recoverable because `RING_MINOR_OVER_MAJOR` *is* the inclination cosine) — and hard-cuts the arc that falls inside the planet's down-sun shadow cylinder (`perp < R_p` on the `along > 0` side), darkening it flat to `SHADOW_DARK`. The cylinder wall (`perp = R_p`) is two straight lines parallel to the light direction, so the darkened arc gets crisp, light-aligned edges — the classic shadow cut, not a radial fade. `LIGHT_Z` places the sun's depth (above ~1 it reads as sitting far behind the scene, casting the shadow forward onto the ring rather than grazing from above); it's plausible, not physical, tracking *which side* the star is on. Hover fills the whole annulus solid white via a per-mesh uniform.

Disc sizing maps a body's `radiusEarth` to a pixel diameter. **Moons** use a plain cube-root compression — `cbrt(radiusEarth) × MOON_DISC_BASE` clamped to `[MOON_DISC_MIN, MOON_DISC_MAX]` — because moon radii are all sub-Earth and read fine under one compressive curve; the moon cap deliberately exceeds the planet floor because big moons cluster around big planets (Ganymede / Titan orbit gas giants), so a max-cap moon always sits next to a near-cap planet in practice — comparing it against the planet floor would be the wrong reference. **Planets** can't use a single curve, because the catalog's radii are bimodal: rocky worlds spread smoothly up to a couple R⊕, then gas giants pile up near ~1 Jupiter radius since electron-degeneracy pressure flattens the mass→radius relation (a half-Jupiter-mass and a ten-Jupiter-mass planet are nearly the same size). A lone cube-root curve renders that pile-up as a near-flat plateau where radius differences vanish into a shared ceiling. So `planetDiscPx` (`layout/row.ts`) blends two radius→size mappings into one monotonic curve: a low-end cube-root mapping that pins Earth to `PLANET_DISC_BASE` and spreads the rocky worlds, and a high-end locally-linear mapping (`PLANET_DISC_GIANT_SLOPE` / `PLANET_DISC_GIANT_OFFSET`) that gives the degeneracy-clustered giant band real per-pixel separation. A `smoothstep` over `[PLANET_DISC_BLEND_LO, PLANET_DISC_BLEND_HI]` hands low→high; a log-sum-exp soft-min then asymptotes the top toward `PLANET_DISC_ASYMPTOTE` (a practical max approached, never clipped) while a soft-max eases the smallest bodies onto `PLANET_DISC_MIN`. No hard clamp anywhere, so a bigger giant always renders at least as large instead of saturating onto a flat cap.

There's no orbit, no zoom, no depth attenuation — the *layout* is static. Its one time-driven element is the cargo-ship overlay.

**Cargo ships.** `ShipsLayer` (`layers/ships.ts`) draws the economy's cargo as a stream of small dots over the bodies — the one animated path in this view, threaded `SystemScene.tick → SystemDiagram.update(now) → ShipsLayer.update(now)` (called each frame, but stepping the dots on a fixed cadence — see below). `EconomyBridge.predictedClusterFlows(clusterIdx)` reads the **speculative next-turn** economy (the cargo the economy is about to dispatch, not what's already in flight — see [the facilities doc](../facilities/README.md)) and hands the layer a set of *lanes*, each classified relative to the viewed system: **internal** (body → body within the system), **outgoing** (a body's export rising off the top edge), **incoming** (an import descending from the top onto its body), and **through** (relay traffic crossing a transit band sideways). The **internal** lanes come from the speculative dispatch plan's same-cluster moves — the intra-system reallocation that resolves with **0 turns of transit** (delivered the instant you hit Next Turn, never aloft) — while outgoing/incoming/through come from the speculative transfer ring (only inter-cluster cargo crosses the void in flight). Each lane emits dots at a rate proportional to its predicted shipped volume; a dot leaves a random point on the source body, follows a quadratic Bézier under a constant-acceleration profile (the ramp anchors only at a body end — an outgoing ship accelerates the whole way out, an incoming one brakes the whole way in, an internal body→body hop does accel-cruise-decel, and through-traffic cruises flat; short hops peak below cruise) and arrives at a random point on the destination body. Travel is screen-normalized, so a dot's journey takes the same wall-clock time on any window size. Dots are re-derived + redrawn every frame (so they track the layout), but their *motion* only advances on a fixed cadence (`SHIP_UPDATE_INTERVAL_MS`) — the journey time τ is held between beats, so positions step on the cadence rather than creeping sub-pixel and snapping at uneven moments (an irregular shimmer, worst on the curved arcs). Body→body lanes bow into an arc (control point offset perpendicular to the chord, signed-random per ordered body-pair and cached so a lane's whole stream shares one curve, the bend reading as a distinct corridor); off-body ends pin to the chord, so outgoing/incoming/through collapse to the straight segment. The endpoint scatter and the arc are ephemeral render state — plain `Math.random()`, not the deterministic sim. The pool is one `THREE.Points` buffer (`snappedDotsMat`, `DynamicDrawUsage`, positions computed on the CPU and rewritten in place — no per-frame allocation) drawn with `depthTest` off and a high `renderOrder`, so dots read as a flat overlay flying over every body without writing depth — they don't rely on z for layering. Through its accel/brake ramps a dot also trails a short yellow exhaust burn — a second in-place `LineSegments` pool, one segment per ramping ship along ±the path tangent (behind while accelerating, out the front while braking, dark at cruise), drawn just under the dots so a dot covers its own burn tip; a 1px hull gets a stubby 1px flame instead of the full length. Each ship also rolls, fixed for its journey, a cruise multiplier (so the stream spreads across a speed band rather than moving in lockstep) and its own **appearance** — a **metallic-sheen color** (random hue held to low saturation + high lightness) and a **weighted 1–3px size** (smaller hulls more common — `SHIP_SIZE_PX_WEIGHTS`), so a lane reads as a stream of individually-hulled ships of mixed size rather than one uniform substance. Color + size ride per-vertex `color` / `aSize` attributes on the same `Points` pool (the `snappedDotsMat` `vertexColors` + `vertexSizes` paths) and are re-uploaded only when the live set changes. Body anchors (center + disc radius) come from a unified `BodyCenterIndex` the diagram assembles each layout from the planet/moon/belt center indices, so a dot can spawn at, scatter across, or target any body kind. A live dot stores only its **normalized journey time** (τ) plus its endpoint *identities* (which bodies, plus a per-end scatter as a fraction of the disc radius) — never absolute coordinates — and maps that time to an arc position through the shared motion profile (`ship-profile.ts`), re-derived from the live layout each frame, so a resize moves the bodies and the dots follow, and a mid-turn flow refresh never re-aims a dot already in flight. The overlay **opens already at steady state**: on view-entry `ShipsLayer.prime` seeds every lane with the in-flight traffic it would carry once running (each ship at a uniform random journey time — correct because, *in time*, in-flight ships are spread uniformly), so a freshly opened system reads as busy rather than filling from empty over a full transit time. The lane schedule is refreshed on Next Turn and after a facility edit; because it reads the speculative next-turn economy it shows the cargo the economy is about to move — a new provider's lanes appear the instant it's built, and the stream never blanks across an edit (the speculative world re-dispatches on every recompute, even when a structural edit dropped the live ring). All cargo-ship tuning lives in `layout/constants.ts` under the `SHIP_*` prefix — emission rate, the wall-clock cross-screen pace and acceleration-ramp duration, the position-update cadence, per-ship speed variance, the per-ship metallic-sheen color band (hue/saturation/lightness ranges), the per-ship weighted size distribution, pool cap, the transit-band height, and the exhaust-burn appearance. The within/from/to/through classification is `src/facilities/flow-class.ts`; see [the facilities doc](../facilities/README.md).

**Fleet (built ships).** `FleetLayer` (`layers/fleet.ts`) draws a system's ready ships — this system's durable `'ready'` ships (see [the game-systems doc](../../docs/game-systems.md)) — as **two opposing formations** in the open field below the planet row, fed via `SystemDiagram.syncFleet`. The ships split by **faction**: the controlled side grids in the left half facing right, every other faction in the right half facing left (`FLEET_MINE_CENTER_FRAC` / `FLEET_THEIRS_CENTER_FRAC`), so the two read as a stand-off across the battlefield. Unlike the cargo overlay it is **not** body-anchored: a ship is a peer of planets, not tied to a disc, so the formations derive from the content rect alone and re-lay-out on resize. Sprites are `Mesh + PlaneGeometry` quads on `makeFleetTriangleMaterial` — a flat faction-tinted triangle whose `uDir` faces it toward the enemy, drawn by the same parity-snapped per-pixel SDF discipline as the star disc — bound by the CPU-rewritten-layer trip-wires (`disableCulling` + `depthTest:false` + a dedicated `RENDER_ORDER_FLEET`). A ship is **pickable**: `FleetLayer.pickAt` runs a radial hit-test over the sprites (pure geometry in `fleet-pick.ts`, a circle around each sprite center regardless of the triangle silhouette) and `SystemDiagram.pickAt` resolves a ship hit ahead of the body z-walk — the fleet is a foreground overlay, so it wins outright. A clicked ship drives the sidebar's read-only ship card (`SystemContext.setShip`, name / class / faction / status); there is no hover rim yet (the material has no outline channel), so the pointer cursor is the only hover affordance. The pool is hard-capped (`MAX_FLEET_SPRITES`, the two sides sharing it) because ready ships have no exit path until movement/combat land. Tuning lives under the `FLEET_*` / `Z_FLEET` / `RENDER_ORDER_FLEET` names in `layout/constants.ts`. Ship *ownership* — the faction model the formations split on — lives in [`src/factions/`](../factions/README.md).

**Hover info card.** `SystemDiagram.pickAt(x, y)` is the picker for the system view. It resolves the **fleet first** — a built-ship hit is a foreground overlay with no z-band (see *Fleet* above), so it wins outright and returns before the body walk — then falls back to the *topmost* body, the one the depth test would have left visible at that pixel, so the cursor and the eye always agree. Resolution is by world z (`bandZ`) at two levels: each pool returns its own largest-z hit (`pickDiscPool` walks all slots, not first-match — one pool mixes discs from many z bands, e.g. a big inner planet's disc reaches into its outer neighbor's slot), then the coordinator keeps the largest-z hit across pools. Both mirror the depth-test winner. This is load-bearing across z bands: each row item (planet/belt + its moons/rings) draws in its own band (`rowIdx · Z_STRIDE`), so a body in a higher band fully occludes a lower band's whole stack regardless of sublayer — a flat layer-priority (or first-match) walk would let a left planet's front moon (or its disc) out-rank a right planet's disc that renders on top of it. Within that body walk, the query order (front moon → front ring → planet → back ring → belt → back moon → star) is only the tiebreaker for the equal-z case (one row item's own stack), mirroring its back-to-front render order. Discs use `(x − cx)² + (y − cy)² < r²`; belts use a bounding-box test; rings use a tilted-ellipse annulus test (inverse-rotate the cursor into the ring's local frame, then check ρ² ∈ [innerR²/outerR², 1]) with a back/front half filter that mirrors the render-order split. The picked body drives two side effects each `pointermove`: `SystemDiagram.setHovered(pick)` flips per-vertex `aHovered` (planets / moons / belt + ring chunks) or a `uHovered` uniform (stars). The disc shader stamps a 1-px white rim when `length(d) > vRadius - 1.0`; the chunk shader inverts the entire chunk to white below 3 px (no room for a rim) and stamps a 1-px white border above. `SystemHud.setHoveredBody(pick, x, y)` updates the `BodyInfoCard` tooltip — a `BasePanel`-based widget that rebuilds its canvas only when the picked body changes, then places itself with a small cursor offset and edge-flip when near a viewport edge. Card rows are scoped to gameplay-relevant character rather than full physical readout: **planets and moons** surface class + temp + pressure + life (when present) + top-2 atmospheric gases with molar fractions (`H2 80%, He 19%`, so Uranus's CH₄ enrichment reads distinct from Jupiter's) + top-2 dominant resources as per-resource abundance rows with 1-5 asterisk ratings (one row per resource — `metals ***** / silicates **` reads as "rich iron, modest rock" faster than collapsing both into a name list that hides whether the world is barren). Ratings bucket the absolute abundance via `ceil(abundance × 5)`, so 1★ ≈ trace and 5★ ≈ near-saturation; the same abundance axis drives the renderer's surface zoning (the primary's Uplands area share + its regolith dilution) so card and disc agree visually. Cloud condensate species and haze aerosol species feed the rendered disc but no longer surface as their own rows — the renderer's blend is the visible signal, and crowding the card with chemistry detail competes with gameplay-relevant fields. **Gas giants, ice giants, and gas dwarfs** drop the resources row entirely since their bulk is out of reach for surface mining, leaving the gas row as the sole extraction signal; **belts** surface AU extent + largest body (km — Ceres-class for shepherded belts, sub-50-km for free-float dust cascades) + shepherd planet (when the belt is dynamically anchored to one) + top-2 dominant resources as per-resource abundance rows; **rings** surface planet-radii extent + their single dominant resource as a per-resource abundance row (a ring is one smeared body — it draws one resource, not two). The underlying resource grid still drives rendered color/alpha and gameplay mining yields — the card just doesn't surface the long form. `pointerleave` on the canvas clears both. Skipped when the cursor is over interactive HUD chrome (the back button or the sidebar), so a tooltip can't appear under what the user is aiming at.

**Body selection + facilities.** Beyond hover, the system view carries a *persistent* body selection (`SystemScene.selectedPick`). A click that misses HUD chrome picks the disc under it and selects the body when it can host at least one facility — eligibility is the facility registry's call (`addableTypesFor(body, [])` from `src/facilities/`), which today means `planet` / `moon` / `belt`; a ring, a star, or empty space clears the selection, and `Escape` clears it before a second press exits the view. Selection reuses the hover rim — `SystemDiagram.setSelected` lights the same 1-px outline, and a body stays lit while it is hovered *or* selected (so moving the cursor off the chosen body doesn't drop its rim), re-stamped after each layout so it survives a resize. A **fleet ship** is the other selectable pick (`FleetLayer.pickAt`): it shares `selectedPick` but has no rim yet (its material carries no outline channel, so `writeOutline` no-ops the `ship` kind) and drives a read-only ship card in the same `SystemContext` — see *Fleet* above. The selected body drives the sidebar's `SystemContext` (`src/ui/sidebar/system-context.ts`) — it lists the body's facilities (each with a remove ✕) plus an *Add* pill per type the body can still host (`addableTypesFor(body, facilities)` — registry predicate plus per-body build cap, so a barren body hides *mining base* and a fully-built one shows none), the clicked pill's type threaded through the `onAddFacility` callback, all painted into the one sidebar canvas and hit-tested by recomputed rects. `SystemScene.pushSelectionToSidebar` composes the body + its game-state facilities into the context and refreshes the sidebar after each change. Sidebar clicks are absorbed (so they never deselect), and the scene reports the sidebar as chrome so the hover tooltip is suppressed over it. The placements persist in the game-state save; the selection itself is ephemeral — a fresh `SystemScene` starts with nothing selected. See [the game-systems doc](../../docs/game-systems.md) and [the facilities doc](../facilities/README.md) for the save + registry models.

**Facility chips.** `FacilitiesLayer` (`layers/facilities.ts`) paints a horizontal row of small square color-chips over every body that carries placed facilities — one chip per facility, the at-a-glance counterpart to the sidebar's facility list (it shows *all* facility-bearing bodies, not just the selected one). Static, with no per-frame work: chips are written at layout (resize) and re-written on a facility edit via `SystemDiagram.syncFacilities` (called from the `onAddFacility` / `onRemoveFacility` round-trip in `SystemScene`, alongside the bridge + sidebar + flow refresh). It reads placements straight from the game-state store (`facilitiesOnBody`, the same accessor that feeds the sidebar), so a resize re-reads and the chips persist. Each chip is a single snapped `Points` vertex (`color` = fill tint, `aSize` = edge length, `aRound` = which corners to round) drawn through `facilityChipMat` (`materials/system-decor.ts`), a purpose-built shader that paints — per integer pixel inside the snapped N×N quad, via the crisp `gl_FragCoord − vCenter` path the star disc uses — a 1-px frame (`FACILITY_ICON_FRAME_COLOR`, a bright setting that pops the tile) with **pixel-rounded corners** (a 45° chamfer the frame wraps), a **1-px inner shadow** on the inner top + left edge (a recessed-tile read), and a **subtle Bayer-dithered diagonal gradient** of a muted fill, slightly brighter top-left to darker bottom-right. The fill color is the registry's (`facilityColor` — one swatch per type on `FacilityDef`, rendered verbatim since ColorManagement is OFF). Chips sit in `addOrder` left-to-right and **overlap by `FACILITY_ICON_OVERLAP` px so a multi-facility row reads as one connected strip sharing a single frame edge per junction** — the `aRound` flag (set per tile in the layer: lone tile rounds all four, a row's end tiles round only their outer end, interior tiles none) keeps the abutting sides square so only the strip's outer ends round. Placement anchors to the unified `BodyCenterIndex`: a row hangs just above a planet/moon's top rim, and centers *on* a belt (which has no single rim). Like the ships, chips run `depthTest:false` at `RENDER_ORDER_FACILITY` (between the bodies and the cargo dots) so they paint over every body but under the traffic. Real icon sprites swap into the fill later — the colored tile is the v1. Geometry/placement tuning is the `FACILITY_ICON_*` / `FACILITY_ROW_MARGIN` / `Z_FACILITY` / `RENDER_ORDER_FACILITY` block in `layout/constants.ts`; the tile's *look* (corner radius, inner-shadow + gradient strengths) is the `CHIP_*` block atop `facilityChipMat`.

**Adding an overlay layer (the decoration pattern).** New visual elements drawn over the diagram — chips, badges, markers, range indicators — follow the same shape as `ShipsLayer` / `FacilitiesLayer`: one object under `layers/` that owns its `BufferGeometry` + material + `Points`/`Mesh`, adds itself to the diagram `Scene` in its **constructor**, and is wired into `SystemDiagram` at three points — constructed in the ctor, its `layout(...)` driven from `SystemDiagram.layout` (handed the `BodyCenterIndex` when it anchors to bodies — those are content-buffer-pixel coords, **y growing upward**, snapped via `snapPx`), and freed in `dispose`. Two trip-wires, both load-bearing and both silent when violated:

- **Build any snapped-viewport material in the layer's constructor, never lazily mid-layout.** `snappedDotsMat` / `snappedLineMat` / `facilityChipMat` — anything that pushes itself into `snappedMaterials` — relies on `ViewportSizer.apply` (first resize) pushing `uViewport` into every *already-registered* material. A material created later — e.g. on the first layout that finally has something to draw — misses that push and keeps `uViewport = (0,0)`; the snap epilogue then divides by it (`ndc = px / uViewport`), sending every vertex to NaN, so the layer renders **nothing at all** until the next resize. Allocate the pool up front (grow on demand) so the material exists before the first `apply`. This one cost real debugging time — geometry, colors, and coordinates can all be correct and the layer still draws nothing.
- **`disableCulling` any mesh whose vertex positions you rewrite from the CPU** — the cached bounding sphere goes stale, and three would frustum-cull the whole pool.

For a flat overlay meant to fly over the bodies, copy the ships' compositing: `depthTest:false` + a `RENDER_ORDER_*` slot above the body bands and below the ships (z then doesn't decide layering — `renderOrder` does). Free the material with `unregisterSnappedMaterial(m)` **before** `m.dispose()`, so the next scene's resize never touches a dead GPU handle.

Star discs are drawn by `makeStarMeshMaterial` — a Mesh-path shader that takes `uCenter` (in buffer-pixel coords) and `uRadius` uniforms and discards fragments outside the disc via `length(gl_FragCoord.xy − uCenter) > uRadius`. No parity-aware snap is needed because the discs are large and their centers are CPU-positioned at integer pixels by the layout pass. Interior shading is a solid `uColor` body with one Bayer-dithered ring of warmer/darker pixels stippled just inside the outer edge — stipple density falls linearly from the rim inward over `INNER_EDGE_DEPTH_PX`, so the fringe tapers into the body rather than terminating on a hard inner border. The fringe color is `pow(hueDir(uColor), INNER_EDGE_SAT_EXP) × uColor × INNER_EDGE_BRIGHTNESS`, where `hueDir` is `uColor` normalized to max-channel = 1; raising it to a power crushes the minor channels while leaving the dominant one untouched — a saturation operator that works for any hue (warm stars cool toward red, cool stars cool toward deep blue, both via the same code path rather than a hardcoded per-channel modulator). The outer halo (`makeStarHaloMaterial`, additive blending, depthWrite off) wraps the disc with a dithered cloud of pixels that thin out with distance: an ordered Bayer threshold gates pixel visibility against a quadratic radial falloff (`pow(1 − t, FALLOFF_EXP)`), so no smooth alpha is ever sampled. The annulus is split into four bands (`HOT_END / WARM_END / EMBER_END`); each band paints `uColor` saturated by a different exponent (`HOT_SAT_EXP / WARM_SAT_EXP / EMBER_SAT_EXP / DARK_SAT_EXP`) so the halo cools through the star's own hue from a near-body color at the disc edge to a deep saturated dark at the fringe. To keep highly-saturated catalog colors (deep blue A/B/O stars, deep red M/BD stars) from colliding with the dithered fringe + halo as a "neon" body, `tuneStarColorForSystemView` in `stars-row.ts` lifts the system-view base color toward white by an amount proportional to color saturation (`max − min`), capped at `SYSTEM_VIEW_SATURATION_LIFT_MAX`. Galaxy view always uses the true `CLASS_COLOR`; the lift only affects the system diagram. Planets and moons render through `makePlanetMaterial` — a `gl.POINTS`-based shader that keeps the parity-aware center snap and `gl_FragCoord − vCenter` disc discard from the perspective stars path, and composes a per-fragment layered stack (bottom to top): surface → cloud → haze → rim. Each layer's alpha is data-driven; total opacity emerges from the composition rather than a mode flip. The disc interior and the atmospheric-loft rim are then lit per-fragment by the in-scene star positions (a dithered colored crescent on the disc, a directional star-colored glow + depth-graded Rayleigh hue shift on the loft) — see "Procedural disc texture" below. The per-body palettes + scalars come from `disc-palette/` (see "Procedural disc texture" below). See the "Pixel-perfect rendering" notes below for the shared snap math.

**Disc / halo split (planets only).** Planets render this material in two passes sharing one `BufferGeometry`: a disc-only pass at `RENDER_ORDER_PLANET` and a halo-only pass at `RENDER_ORDER_PLANET_HALO` (after front rings and front moons). Mode is selected via `makePlanetMaterial(scale, 'disc' | 'halo')`, which gates the disc-interior and halo branches with `#define DISC_ONLY` / `#define HALO_ONLY`. Splitting is load-bearing for the cross-row blend: the halo extends a few pixels past the disc and would otherwise blend against background where it overhangs a left neighbor's front ring or moon (those layers haven't drawn yet at `RENDER_ORDER_PLANET`). Depth test on the halo pass keeps R's halo from painting over R's own front ring/moon (higher z) and rejects L's halo where R's disc already wrote depth. Moons keep the single-pass `'all'` mode — their halos are small enough that the analogous bug is invisible in practice.

## Procedural disc texture (planets + moons)

The planet/moon disc shader (`makePlanetMaterial` in `materials/planet.ts`) composes four data-driven layers per fragment — surface, cloud, haze, rim — and `disc-palette/index.ts:buildDiscPalette` derives the per-layer palettes and scalars from each body's data. A body with thin patchy H₂O cloud over a visible rocky surface (Earth) and a body with full-coverage banded NH₃ over a hidden gas-giant interior (Jupiter) run through the same shader path; only the layer alphas differ.

The detail lives in three module headers — read them when working on the corresponding concern:

- **Palette derivation** — the `disc-palette/` modules (`index.ts` surface palette + assembly, `atmosphere.ts` haze/rim/Rayleigh, `ocean.ts` ocean color, `lava.ts` molten drives). One-primary-colour surface model: `buildDiscPalette` derives a single primary colour from the body's dominant resources — the primary resource's archetype hue diluted toward `REGOLITH_BASE_COLOR` by the regolith share `1 − a0` (so a rich world keeps its resource hue and a poor one washes to barren grey) — and hands it to the shader as slot 0, with the raw secondary resource hue in slot 1. Abundance is carried by AREA, not colour: the primary abundance `a0` (`vWeights.x`) splits the disc into an Uplands resource patch and a Lowlands regolith remainder (clamped off the extremes by `ZONE_AREA_FLOOR` so neither vanishes), which biases the shader's elevation field high/low; the secondary abundance `a1` (`vWeights.y`) is NOT an area — it sets the elevation level the secondary hue floods the low ground up to. Surface relief roughness is also derived read-side here, in `terrainRoughnessFor`: a `reliefBands` terrace count (relief depth) and a `granularity` feature-size factor, composed from stored physics (surface gravity from mass/radius, `surfaceAge`, `tectonicActivity`, erosion proxies — pressure / liquid / warm-ice relaxation) rather than stored as fields. World-class colour re-enters only as a flat-fill fallback when a body carries no resource signal at all. Plus the four-category haze contributor blend (bulk gases × Rayleigh × aerosols × dust, each gated by `log10(P+1)` and a global category scale), the rim color merger, the per-gas limb Rayleigh scatter color + Rayleigh-fraction strength (`scatteringRimFor`, feeding the lit-rim hue shift), the two-layer color tables (`GAS_COLOR` for gas-phase + photochemistry aerosols, `CONDENSATE_COLOR` for ice/frost appearance), and the six-pathway physically-parameterized ocean color derivation (`oceanColorFor`: solvent species → CDOM → pigment → sediment → sky reflection → stellar SED) so close-analog bodies get distinguishable surface-liquid hues.
- **Shader composition** — `materials/planet.ts` header on `makePlanetMaterial`. Layer order, sphere projection, worley/voronoi cells in `(lon, lat)`, surfaceAge- and `shellFraction`-keyed crater + ejecta-ray + linea sub-passes (the crater reveal lerps buried regolith ↔ bright excavated ice by `shellFraction`, and linea-crack density rides it), the continuous ice-cap model (one latitude threshold at the `iceCoverage`-derived snow line — caps grow from the poles to a wholly frozen disc as coverage rises, with no cap-vs-global regime split and no temperature re-derivation, since `iceCoverage` is a pure function of the procgen `iceFraction`; the margin is shaped by multi-octave value-noise lobes into ragged fingers/bays and feathered by a cap-proportional, snow-line-centered Bayer-stipple frost fringe, and an ice shell ages under a mottled sublimation dust lag — multi-octave value-noise coverage driven by an `ICE_WEATHER_FLOOR`-lifted surface age with a gentle equatorial lean, tinting the ice toward the body's own primary-resource colour (dirty ice, not black rock) and gated by `shellFraction` (the continuous bulk-ice fraction) so a bulk-ice world darkens while a rocky snowball keeps bright snow that only craters dark), multi-deck cloud compositing with coverage rents revealing the next-deeper deck (and on no-surface bodies a synthetic base deck — atm column lifted slightly toward white — as the foundation under chemistry decks), the firing/firing + firing/non-firing Bayer-dither edge cases, the per-deck haze pre-tint, the wind-driven cloud morphology spectrum (slow-wind decks stretch into zonal streaks and pick up a sparse vortex-eye swirl — `cloudSwirl` — that winds them into scattered cyclones over the bands, while fast-wind decks band into latitude strips and the swirl fades out by `bandness` so gas giants stay straight), the 1-px-down cloud drop shadow (`CLOUD_SHADOW_*`, the deck stack reprojected one screen-px up and re-tested, comparing topmost-deck indices so an upper deck shadows the lower deck exposed through a rent below it, not just the surface) darkening the layer beneath each cloud's lower edge, the star-lit atmospheric-loft rim halo (directional gas × starlight glow with a half-lambert wrap past the terminator + gated white forward-scatter tip, then a depth-graded hue-only Rayleigh shift toward the body's per-gas scatter color), the per-fragment crescent body lighting keyed off the in-scene star positions, parity-aware center snap. The crescent's width and the loft-rim's arc spread both derive from a single shared light-depth knob (`LIGHT_Z_BIAS` in `chunks.ts`, with the rim's `RIM_GLOW_FOCUS` derived from it), so pushing the star deeper behind the bodies narrows both in lock-step. The surface block paints a terraced elevation field. Elevation is multi-octave relief noise (its wavelength scaled by the per-body `granularity`, so feature size varies) plus a resource-topology bias: an abundance-weighted jittered worley tessellation claims each patch as Uplands (the resource patch, area ≈ `a0`, biased high) or Lowlands (regolith remainder, biased low), so the relief comes from the independent noise while the resource patch still reads as bright high ground regardless of how rich the world is. That elevation is quantized into `vReliefBands` discrete terraces, each a fixed `SURFACE_SHADE_STEP` of value off the primary colour (lighter uphill, darker downhill, mirrored about the mid terrace) — the per-step contrast is constant and the band COUNT is the relief depth (1 ≈ flat, 6 ≈ rugged), with a sub-pixel ordered dither fraying each terrace contour; a crater/linea excavation always exposes the deepest terrace's buried regolith. The secondary resource then floods the low ground as a topographic stain (everything below an `a1`-set elevation level takes a constant partial overlay so the terraces blend through, the waterline dithered for a frayed edge). Liquid floods the same elevation field below `vLiquidFrac`, and the submerged terrain reads as a continental shelf taken straight from the topo (solid highlight in the shallows thinning to a depth-proportional dither at the shelf break, so shelf width follows the seabed slope rather than a fixed ring) so large oceans don't read as flat fills.
- **Chemistry that feeds it** — `procgen.mjs:hazeContribution` per-species gates (THOLIN, NH4SH, CHROMOPHORE, SALT, H2SO4, SULFUR, SILICATE) and `procgen-priors.mjs:CONDENSABLES` (per-species condensation rows with peaked T windows, altitude offsets, precursor gates — including the hot-Jupiter reflective IRON deck and the ultra-hot dark TIO absorber). Procgen owns the chemistry; the renderer paints exactly what's emitted.
- **Gas-giant thermal appearance** — a giant's look tracks its cloud-top temperature, which folds intrinsic Kelvin-Helmholtz heat into irradiation (the combined T_eff in the Filler, so a young/massive giant reads hot far from its star). As T climbs the atm column collapses toward a dark absorber (`hotGiantColumnDarkening` — the <10%-albedo hot-Jupiter look), the reflective IRON/SILICATE and dark TIO decks above replace the cold ammonia/methane chemistry, and past the incandescence floor the body self-emits a deep dull red-orange glow (`giantEmissionNormFor` → the shader's `emberRamp`, painted as a smooth global ember under the decks and revealed through their rents). Reflectance-dark yet self-luminous — no stored albedo, no class branch; it all rides the one temperature.

Two architectural commitments to remember:

- **Visual weight ≠ molar fraction.** Every haze contributor weights species by `GAS_POTENCY[gas]`. H₂/He/N₂/Ar are calibrated near-zero (essentially transparent in visible — only Rayleigh contributes via `SCATTERING_POTENCY`); CH₄ sits highest among atm species (Uranus/Neptune cyan); aerosols sit at the cloud-former level. Without this, an 80%+ H₂/He bulk would wash trace absorbers to cream.
- **Multi-color character emerges from coverage rents, not in-deck palette mixing.** Each cloud deck has exactly one color (`cloudDeckPalette(body, gas)`). Jupiter's NH₃ rents to expose NH4SH below; Uranus's sparse CH₄ cirrus rents to expose the deep H₂O deck plus the atm column. One deck = one color is load-bearing — it keeps the shader simple and lets the data drive variety.

Body brightness is **not** modulated by a stored albedo scalar — reflectance emerges from resource composition + ocean composition + ice + biome + cratering + linea + terraced relief + the atmospheric layers above. By the same discipline, a body's **type** isn't stored either — and isn't even collapsed to one: there is no `worldClass` field and no single classifier. The disc's base color is the resource mineralogy (plus ocean / biome / atm-column / belt-icyness), and the label is composed at read time from the multi-axis `body-traits` predicates — so appearance and label both follow the simulation rather than a hand-set category, and a multi-axis world keeps every axis.

## Pixel-perfect rendering — the load-bearing constraints

**The pixel-crisp look is the committed visual identity** — not a stepping stone toward a softer aesthetic, and deliberately distinct from the organic-CG genre peers (Ascendancy, Master of Orion 2). If a future feature seems to want a gradient, an anti-aliased stroke, sub-pixel positioning, or DOM-rendered text inside the canvas, the answer is to find a pixel-crisp way to express the same intent (dithering, alternating-row tints, halo'd labels, palette swaps) — not to soften the rules. The painter primitives in `src/ui/painter.ts` and the theme tokens in `src/ui/theme.ts` encode this commitment; treat them as constraints, not defaults to grow past.

The whole "pixel art" look depends on a stack of choices that all have to stay consistent:

1. **`renderer.setPixelRatio(devicePixelRatio / N)`** where N is an integer in {1, 2, 3, 4} chosen at runtime by `RenderScaleObserver` (in `src/render-scale.ts` — a shared root module the scene's `ViewportSizer` and the HUD's resolution radio both read) to land closest to a 72-DPI visual size for the current `devicePixelRatio` — typically 3 on retina (DPR=2) and 1 on a 1080p desktop (DPR=1). The render buffer is sized so each render ("env") pixel becomes N×N physical pixels after the browser's `image-rendering: pixelated` nearest-neighbor upscale. The user can bias N via the **Resolution** radio (`Low` = +1 chunkier, `Medium` = auto, `High` = −1 sharper); `effectiveScale(auto, pref)` clamps to {1..4}, and the panel disables options that would clamp to a no-op (e.g. `High` on a DPR=1 display, where auto is already 1). DPR boundary crossings (browser zoom, monitor swap, OS scale change) re-fire the observer so the auto N stays current. Increasing N makes the look chunkier and reduces fragment work by 1/N². Critically, **all pixel-aware shader work must use `renderer.getDrawingBufferSize()` — NOT `window.innerWidth/Height`** — because the buffer is now smaller than the CSS viewport. `scene.ts` caches these as `bufferW`/`bufferH` in `resize()` and threads them into `setSnappedLineViewport`, `StarPoints.setPxScale`, `Labels.update`, and `Hud.resize`. Pointer math (raycast NDC, HUD click coords) uses cached `cssW`/`cssH` rather than `window.innerWidth/Height` for the same reason — the canvas may be a few CSS pixels smaller than the window after the integer-multiple rounding (next point).
   **Integer-multiple sizing is load-bearing.** The browser's nearest-neighbor upscale is only exactly N:1 when the target physical pixel count is divisible by N. `resize()` rounds CSS×DPR (the target physical dimension) DOWN to a multiple of N before calling `setSize`, then derives CSS and buffer dimensions from that. Without this rounding, a non-divisible window (e.g. 1366px wide at DPR=2 = 2732 physical px = 911 buffer × 2.999 upscale) gets one buffer column every ~911 columns squashed into 2 physical px instead of 3 — visible as a column of mangled pixels in any label that happens to sit on top of it, with the artifact "following" labels as the camera rotates and they cross fixed bad columns. Cost: up to N-1 physical px of black bezel on the right/bottom (invisible against the dark scene + matching body bg).
2. **Pixel-snapped line shader** (`snappedLineMat` in `materials/galaxy.ts`) — the vertex shader rounds each projected vertex to the nearest integer screen pixel before rasterization. Eliminates sub-pixel shimmer on thin lines. Used for grid arcs, axes, the galactic-centre arrow, and the solid variant of droplines. A sibling `snappedDotsMat` does the same for `Points`, parity-matching the snap to the requested point size (odd sizes snap to a pixel center, even to a boundary) so a dot rasterizes crisply rather than straddling — the dotted dropline variant at its single-pixel default, the system-view cargo-ship dots at a per-ship weighted size (`SHIP_SIZE_PX_WEIGHTS`, via a per-vertex `aSize` attribute that also drives the per-vertex parity snap).
3. **Stars shader** (`makeStarsMaterial`) — `gl.POINTS` with a procedural circle in the fragment shader (no texture sampling, no AA fringe). The vertex shader rounds size to the nearest integer pixel count (so zoom transitions step 2→3→4→5…) and snaps the projected center to the pixel grid using a **parity-aware** snap: even sizes snap to a pixel boundary (integer window coord), odd sizes to a pixel center (half-integer). The snapped center is passed to the fragment shader as a varying `vCenter`. The fragment shader then computes its pixel-grid offset directly from `gl_FragCoord.xy - vCenter` — `gl_FragCoord.xy` is always integer+0.5, and `vCenter` is integer or half-integer, so the difference lands at clean pixel-spacing offsets symmetric about both axes by construction. **Don't use `gl_PointCoord`** for the discard test: its sub-pixel precision is implementation-defined and produces visibly asymmetric discs on some GPUs when the point center sits at sub-pixel positions. The discard threshold is the true Euclidean radius (`length(d) > vRadius`) so sizes 1/2/3 render as full squares and size 4 onward starts dropping corners — the natural pixel-disc progression. The pixel-snap math runs **after** the perspective divide (`clip.xy / clip.w`) so it works identically under ortho and perspective projection.
4. **Label overlay pass** (`Labels` in `labels.ts`) — labels are rendered in a second ortho pass at 1 unit = 1 buffer pixel, the same scheme as the HUD, rather than as 3D Sprites in the main scene. Each frame the cluster primary's world position is projected by the **main** camera; the result drives a `Mesh + PlaneGeometry` placement in the overlay scene, with the top-left corner snapped to an integer buffer pixel so every texel renders. Constant on-screen size keeps typography stable while the depth-attenuated stars do the depth-cueing work — depth-scaling labels on top of depth-scaling stars would just make distant labels illegible.
5. **HUD** (`MapHud` in `src/ui/map-hud/`, `SystemHud` in `src/ui/system-hud/`) — third ortho pass at 1 unit = 1 buffer pixel, rendered after the main scene and label overlay with `autoClear` toggled off. Geometry is `Mesh + PlaneGeometry + MeshBasicMaterial` (the `Widget` base in `src/ui/widget.ts`) so positions and sizes are integer pixel counts that match the rest of the scene's grid. The settings trigger is a hamburger glyph in the sidebar header (`Sidebar.onSettings` → `MapHud.toggleSettings()`); the panel anchors just left of the sidebar, with its close-X at the panel's own top-right corner. The settings panel and the sidebar extend `BasePanel`: a single canvas texture rebuilt on state change (toggle flipped, hovered row changed, selection changed) — cheaper than maintaining one texture per row state because each panel is small and rebuilds run only on user input, not per frame.

If you add new scene geometry, route it through `snappedLineMat` for lines and the existing point-shader pattern for sprites — don't introduce vanilla `LineBasicMaterial` or `PointsMaterial`, they'll shimmer.

## Color management is OFF

`app-controller.ts` runs `ColorManagement.enabled = false` at module load and sets `renderer.outputColorSpace = LinearSRGBColorSpace` on the shared `WebGLRenderer` in the constructor. This is intentional and load-bearing.

The whole project's palette is hand-picked sRGB hex values (`0x1e6fc4`, `#5ec8ff`, etc.) intended to render at *exactly* those values on screen. With Three.js's default color management, two parallel paths (shader uniforms via `new Color(0x...)` vs canvas-texture pixels via `fillStyle = '#...'`) get different sRGB↔linear conversions and end up rendering at *different* on-screen colors — visible wherever a canvas-rendered text label sits next to a shader-drawn grid line, dropline, or arrow at the same hex. With management off, every hex value is the displayed value end-to-end, and there's no lighting math to break.

Don't re-enable color management without auditing every call site that mixes `new Color()` (in shaders) with canvas-rendered text textures.

## Depth-attenuated star sizing

Under perspective, the stars vertex shader scales each disc by a depth factor derived from view-space distance (`REF_DIST`). REF_DIST anchors the curve at the value the per-class table sizes were tuned against; it's intentionally decoupled from `DEFAULT_VIEW.distance` so the default framing can be tweaked without rescaling every disc.
- Raw factor: `REF_DIST / dist`. At `REF_DIST` away the factor is 1 and the star renders at its `pxSize` value.
- **Asymmetric:** close-up side is cube-root-compressed (`pow(rawScale, 1/3)` when `rawScale > 1`), zoom-out side is linear. Linear close-up growth eats the screen — at orbit 5 ly a focused class-G star wants 10× growth and ends up dominating. Cube-root compression preserves the per-star ratio but tames absolute growth — orbit 5 → 2.15×, orbit 4 → 2.32×. The exponent (1/3) is the tuning knob: smaller = flatter close-up, larger = more growth. Zoom-out stays linear so distant fields shrink at the natural rate.
- Floored at 2 px so the smallest dwarfs stay visible at zoom-out. **No upper bound** — an upper clamp on the final size flattens the brightest entries into a single blob the moment any of them hit the cap.
- Computed per-vertex from `modelViewMatrix * position`, so it picks up both camera-to-target distance and each star's offset from the focus naturally.

White dwarfs (`pxSize ≈ 3`) hit the size-2 floor first as their depth crosses ~1.5 × `REF_DIST`.

## Star pivot-dim (local-focus brightness)

Stars outside the user's current point of interest are dimmed toward `FADE_MIN` brightness so dense starfields read as a focused local volume rather than a wall of equally-bright dots. The dim is computed per-vertex in the stars shader and applied by multiplying `vColor` toward black — **not via alpha**, because stars render `transparent: false, depthWrite: true` for the "closer stars occlude farther ones" guarantee, and going transparent would break attribute-order rendering in dense fields.

Three signals combine:

- **Pivot ramp.** Per-star world-distance to `view.target` (the orbit pivot), keyed to `PIVOT_FADE_NEAR / PIVOT_FADE_FAR` from `cluster-fade.ts` — the same thresholds the cluster label fade uses, so a star and its label dim together at the same distance. (The cluster-fade module's `CAMERA_FADE_*` thresholds are intentionally NOT consulted here; see below.)
- **Zoom-distance scaling.** A per-frame `uDimAmount` uniform driven CPU-side from `view.distance`: full effect when zoomed in, smoothly off when zoomed out. Bounds are `STAR_DIM_FULL_BELOW` and `STAR_DIM_OFF_ABOVE` (also in `cluster-fade.ts`). At the default orbit radius the effect is at full strength; past `STAR_DIM_OFF_ABOVE` the dim is gone entirely so a zoomed-out player sees every star at its natural brightness — the galaxy "opens up."
- **Selection / candidate bypass.** A per-star `aClusterIdx` attribute lets the shader compare against `uSelectedCluster` and `uCandidateCluster` uniforms; matching stars render at full brightness regardless of pivot distance. Mirrors the yellow-label promotion in labels.ts so hovering a far-away star isn't an awkward "the dot dims while the bracket and label pop on top." Hover wins via the same candidate slot the brackets read.

**Why orbit-distance scaling and not a per-star camera ramp** like the labels use: when zoomed all the way out, every star is far from the camera, so a per-star camera ramp would never re-brighten on zoom-out — it would pin even the nearby stars dim. The label fade tolerates this because hiding distant labels at zoom-out is desirable (the waymarker ramp brings back a curated few); for stars the desired behavior is the opposite, so the camera signal is reinterpreted as an orbit-radius "is the effect even active" gate.

`makeStarsMaterial` injects `PIVOT_FADE_*` and `FADE_MIN` as `const float`s baked from the JS imports at material construction; `uPivotWorld`, `uSelectedCluster`, `uCandidateCluster`, and `uDimAmount` are updated each tick by `StarPoints.setPivot / setSelectedCluster / setCandidateCluster / setDimAmount` called from `StarmapScene.tick`. The pivot uniform is kept distinct from `uFocusWorld` (which doubles as the "snap to NDC zero" key for the focused-vertex pixel-snap fix) so the two semantics don't get accidentally coupled if either gets retuned.

The effect is starmap-only; `SystemScene` doesn't apply it.

## Cluster label visibility

When the master `show labels` toggle is on, cluster labels are gated by **two independent distance ramps** that multiply into a final opacity. Either FAR threshold hides the mesh outright (skipped, not drawn at zero alpha). Both thresholds live in `src/scene/cluster-fade.ts` so labels and drop-lines stay in lockstep — a tweak in one place propagates to both consumers.

- **Pivot ramp** — primary's distance to `view.target` (the orbit pivot). `PIVOT_FADE_NEAR`, `PIVOT_FADE_FAR`. Scopes the visible label set to the user's current point of interest.
- **Camera ramp** — primary's distance to the camera. `CAMERA_FADE_NEAR`, `CAMERA_FADE_FAR`. CAMERA_FADE_NEAR is deliberately set above PIVOT_FADE_FAR plus a "reasonably close" orbit radius so at close zoom every label that survives the pivot gate is also well inside the camera bubble — only the pivot gate effectively fires. As orbit distance grows past CAMERA_FADE_NEAR, stars exit the camera bubble and labels dim regardless of pivot.

A third **waymarker ramp** runs in parallel for a curated list of bright, well-known stars (`WAYPOINT_STAR_IDS` in `data/stars.ts` — Sol, Rigil Kentaurus, Sirius A, Procyon A, Altair, Vega, Arcturus). Polarity is *reversed* from the two ramps above: keyed to **the camera's distance from Sol** (`camera.position.length()` — Sol sits at origin), waypoint labels stay invisible below `LABEL_WAYPOINT_HIDE_BELOW`, fade in linearly, and are fully visible at `LABEL_WAYPOINT_SHOW_ABOVE`. Camera-from-Sol — rather than orbit distance — surfaces waymarkers whether the user got "far from home" by zooming out *or* by panning the pivot far from Sol while still zoomed in. The waypoint and per-label opacities combine via `max()` each frame, so a waypoint inside the pivot bubble doesn't blink out between the regular `PIVOT_FADE_FAR` / `CAMERA_FADE_FAR` thresholds and HIDE_BELOW. Effect: as the camera leaves Sol's neighborhood, every label disappears *except* this small set of named anchors — navigation landmarks for orienting in unfamiliar territory.

Selection and candidate (which subsumes hover — see "Cluster brackets") **bypass all three ramps and the master `show labels` toggle**, so the yellow variant is always visible. Both states are first-class focus state — "what's selected" and "what spacebar would select" — not environmental decoration. With labels off, the yellow promotion is the only feedback that pointing at a star (or panning past it) registered. Waypoint stars still respect the master toggle when in their plain (non-candidate, non-selected) state — turning labels off hides everything plain, including waymarkers.

## Range rings + drop-lines (selection-driven)

Both the rings (in `Grid`) and the per-cluster drop-lines (`Droplines`) are gated on the current selection. With nothing selected the entire subsystem is hidden — the catalog reads as plain stars, no chrome. Click a cluster and:

- The grid `Group` (rings at radii 5/10/15/20 ly + cross axes + galactic-centre arrow) translates onto that cluster's COM and becomes visible. The arrow continues to point galactic +X, just from the new anchor — "from here, that way is the centre."
- Every other cluster's drop-line lights up, terminating at the selected cluster's altitude (`STAR_CLUSTERS[selected].com.z`). Visualizes other systems' Z offsets relative to the one under inspection.
- The selected cluster's own drop-line collapses onto the plane (`dz = 0`) and is hidden by `DROPLINE_DEGENERATE_DIST` — no zero-length pin pointing at itself. Sol gets a real pin whenever a non-Sol cluster is selected.

Deselect (`ESC`) → everything fades out and hides again.

### Sequential per-ring expand/collapse on selection change

Selection changes don't pop — the grid choreographs ring visibility so the frame settles into the new position rather than appearing all at once:

- **Expand** — innermost ring fires first, then the next two rings step outward, and the outermost ring + cross axes + galactic-centre arrow share the final step so the +X arrow caps the frame at full extent. Stagger is `RING_STAGGER_EXPAND_MS` per step, so the four steps settle in sequence. Each ring (and the axes/arrow group) is a discrete on/off visibility flip — no per-element opacity ramp.
- **Collapse** — reverses the order so the outermost ring (paired with axes + arrow) is the first thing to disappear, working back to the innermost. Faster stagger (`RING_STAGGER_COLLAPSE_MS`) because the user has already moved their attention elsewhere.

`Grid` owns the choreography end-to-end. Public surface is `setSelection(com | null)` + `update(now)`; the scene routes selection but doesn't drive any per-tick animation state.

Two internal frames (A/B) let an old selection's collapse run concurrently with a new selection's expand on a swap — both render together for the overlap window. On a rapid third-click while both frames are mid-animation, the older frame is snapped to its terminal state and reused for the fresh expand; the newest selection always gets a clean expand from step 0 while the most-recent previous selection keeps its collapse running. Re-passing the active selection's exact COM is a no-op (avoids restarting the animation on a re-click).

Drop-lines currently snap on/off with selection state (`setFade(0)` on deselect, `setFade(1)` on select) rather than ramping in lockstep with the rings — the cross-fade state machine that previously coupled them lived in `scene.ts` and was retired with this split. Restoring grid/dropline lockstep is a follow-up.

### Per-drop visibility

`Grid` is just a translated `Group`; selection writes `group.position` and toggles `group.visible`. One pin per cluster, anchored at the cluster's mass-weighted **center of mass** (`StarCluster.com`, baked into `catalog.generated.json` by `buildClusters` in the build script). Non-primary cluster members — Sirius B, Alpha Cen B, Proxima, the Gliese 570 BC pair, etc. — share the cluster's pin rather than getting their own. The COM (rather than the primary's position) makes a binary/triple read as one system whose pin emerges from the geometric middle of the ring rather than from one of its members. Two flavors exist per pin: a **solid** `Line` (one full-length segment) and a **dotted** `Points` whose vertices live in a pre-allocated `MAX_DOTS_PER_PIN` buffer (`DynamicDrawUsage`). Z-values are rewritten in place when the selection plane shifts, at fixed world-Z intervals (`DROPLINE_DOT_PERIOD_LY`), and `setDrawRange` slices the active count — avoids reallocating attributes on every selection change. Each frame `Droplines.update()` picks solid if the COM is on the same side of the plane as the camera, dotted if on the far side (`camera.position.z >= planeZ`).

Visibility is per-drop, not via `group.visible`. Composed gating, in this order:
- **Selection gate** (the outer one): no selection → every drop hidden, early return.
- **Global fade gate**: `globalFade <= 0` → every drop hidden. Today the scene snaps `globalFade` between 0 and 1 with selection state, so this is effectively a redundant guard alongside the selection gate; the multiplier is preserved for the planned restoration of grid/dropline lockstep, where it will once again ramp.
- **Master HUD toggle** sets `masterVisible`. When off, only the hovered cluster renders its pin (the selected cluster's own pin is degenerate, hidden anyway).
- **Hover** (pointer over any star in the cluster) **always** shows that cluster's pin at full opacity, bypassing the master toggle and fade ramps (still subject to `globalFade`).
- **Pivot + camera distance fades** apply to every other rendered pin, keyed to the cluster *primary's* position (not the COM, so the pin and its label flip in/out together at the same camera distance). Thresholds live in `src/scene/cluster-fade.ts` (shared with labels) — fade is keyed to `view.target` (the orbit pivot), not the locked rings, so "what's near where the camera is currently looking" still applies even after the user pans the pivot away from the selection.

World-space dotting is the load-bearing choice. The previous screen-space dashed shader used a single global gap-scale uniform driven by the camera's orbit radius — fine for the focused dropline, but every other dropline got the same scale, so distant pins ended up with absurdly stretched gaps (or one orphan dash) any time the user zoomed in close to a near star. Baking dots as actual vertices at fixed world-Z spacing lets perspective do the scaling per-line: distant pins compress, near ones stretch, no per-frame uniform plumbing needed. Dot count along a line scales with `|com.z - planeZ|`, so longer pins visibly carry more dots than shorter ones — a real depth cue rather than a synthesized one. The trade is sub-pixel aliasing at extreme distance (period < 1 px), accepted as graceful degradation.

The choice of `Points` over `LineSegments` for the dotted variant is a deliberate simplification: each dot is one vertex with `gl_PointSize = 1`, snapped to a pixel center. No endpoint pairs, no risk of zero-length segments dropping at extreme zoom, and the GPU's point-sprite fast path is cheaper than line rasterization for sub-pixel-thin output.

Materials are cloned **per drop** (a solid + a dotted ShaderMaterial per cluster) so each pin can carry its own `uOpacity` for the fade ramps. Both solid and dotted are alpha-blended; with one pin per cluster and clusters at non-coincident COMs there's no opacity-stacking concern in practice.

Stars themselves are also rendered opaque (`transparent: false, depthWrite: true`) so closer stars correctly occlude further ones — without `depthWrite`, stars in a single `Points` geometry would render in attribute (catalog) order, ignoring camera-relative distance.

## Focus-point marker

A small ring at `view.target` plus an optional dropline down to the selection plane (`FocusMarker` in `src/scene/focus-marker.ts`). Renders whenever the pivot has been panned past a small lateral threshold off the nearest "anchor" cluster — the selection COM when a cluster is selected, the nearest cluster COM otherwise. Hidden by default on initial load (pivot on Sol) and whenever the pivot sits on/near a star; fades in linearly as the user pans into empty space between systems. The ramp is a pure function of `|view.target − anchor COM|`, so the marker tracks frame-by-frame with no animation state.

The dropline portion exists only when a cluster is selected (that's the only state where a plane exists to drop to); the ring renders alone otherwise. The dropline reuses the same `DROPLINE_DOT_PERIOD_LY` spacing and same camera-side-of-plane solid/dotted swap as the per-cluster drop-lines (`droplines.ts`) so the depth language stays uniform across the scene. The ring is colored to match the grid rings so it reads as a small companion to them rather than a different element class.

Suppressed entirely during the focus glide — while `view.target` is in transit toward a newly-selected cluster's COM, the "where am I looking" hint would just trail the camera as it zooms in and read as noise. `StarmapScene` threads its `focusAnimating` flag into `FocusMarker.update()` for this check.

## Cluster brackets — selection + candidate

Yellow corner brackets enclosing a cluster's rendered-disc bbox live in `src/scene/cluster-brackets.ts` (`ClusterBrackets`). Two instances render simultaneously into the labels overlay scene:

- **Selection brackets** (`style: 'arms'`) — full L-corner reticle around the currently-selected cluster. Anchored on the cluster's COM (which is exactly where `view.target` parks after focus completes) and sized as a square large enough to enclose every member's rendered disc — single-member clusters collapse to a tight box, binaries/triples grow symmetrically around the COM. Anchoring on the COM rather than the per-frame member bbox midpoint pins the bracket to the same NDC-(0,0) short-circuit the shared `projectWorldToBuffer` (in `project-buffer.ts`) applies, so sub-pixel FP noise in the matrix math can't twitch the reticle 1 px laterally while the camera orbits. Cleared when the selection clears.
- **Candidate brackets** (`style: 'dots'`) — single-pixel corners (same color, same brightness, same corner positions as the selection arms) around the *candidate* cluster. The dot corners sit exactly where the arms would, so promoting a candidate to selection grows arms outward from the same dots with no positional shift.

**Only one candidate at a time.** The candidate slot is filled by, in priority order:

1. **Hover** — pointing at a star promotes its cluster to candidate. Independent of `focusAnimating` (cursor location is real regardless of camera motion).
2. **Focus-proximity** — when nothing is being hovered, the nearest cluster to `view.target` fills the slot once the pivot has been panned past `FOCUS_MARKER_NEAR` off it. Suppressed during the focus glide (pivot is in transit, not parked off a star) and when the nearest cluster IS the current selection. `FOCUS_MARKER_NEAR` is shared with the focus marker so both indicators turn on/off together as the user pans off a star.

When hover ends, the candidate falls back to whatever the focus-proximity branch yields (or nothing). Both branches honor "candidate != selection" — no point bracketing what's already selected. Visibility is binary — snap on / snap off, no fade ramp.

The same candidate index drives **three** consumers each tick: candidate brackets (dot corners), labels (yellow text promotion + fade-bypass — see "Cluster label visibility"), and the spacebar handler (F is bound separately — see "Input").

Nearest-cluster lookup is centralized in `nearestClusterIdxTo(x, y, z)` in `src/data/stars.ts`, backed by a static 3D k-d tree over `STAR_CLUSTERS` keyed on COM (`src/data/kdtree.ts`). `StarmapScene.tick()` runs it once per frame and shares the result with `FocusMarker` (anchor when nothing is selected) and the candidate-bracket gating. The k-d tree itself is built fresh at module load over the precomputed clusters — derivation work (cluster detection, coincident-set expansion) has already happened in `build-catalog.mjs`, so the runtime only pays the tree construction cost.

## Input

Input is split between two modules. `InputController` (`src/scene/input-controller.ts`) owns every pointer/keyboard listener, classifies the gesture (orbit drag, pinch zoom vs pinch pan, click vs drag, double-click, touch long-press, held-key set), and dispatches high-level intents through an `InputHandlers` callback bundle. `StarmapScene` implements those handlers — view-state mutation (orbit/pan/zoom/held-key physics), selection logic, focus-glide, and the per-tick raycast for hover. Pure camera math (`applyOrbitDelta`, `applyTouchPan`, `zoomBy`) and selection routing stay in the scene; the controller never reads or mutates view state directly.

The behaviors below describe the user-facing intent — which gesture maps to which camera/selection action.
- **Mouse / pen drag** (any button) = orbit (yaw/pitch). Always orbits regardless of the touch-input setting — single-button mice don't have a clean equivalent of two-finger gestures.
- **Single-finger touch drag** = orbit by default; pans the camera target along the camera's screen-aligned right/up axes when "Pan with single touch" is enabled in the settings panel. The setting is persisted via `localStorage` (see `src/settings.ts`) and read fresh per gesture, so toggling takes effect on the next pointer event.
- **WASD** = pan the orbit pivot parallel to the galactic plane (z=0). W/S move along the yaw heading (the camera's view direction projected onto the plane), A/D strafe perpendicular to it. Pitch is ignored on purpose: looking down at a star and pressing W glides over it instead of plunging into it. Camera and target translate together so the orbit radius is preserved. Pan rate scales with `view.distance` so the screen-space movement rate is consistent across zooms.
- **Z / X** = sink / lift the orbit pivot along world up (the galactic plane normal). Same camera-follows-target translation and `view.distance`-scaled rate as WASD, so the view drops below or rises above the plane without changing orbit radius.
- **Q / E** = orbit left / right around the current pivot (yaw rate is constant in radians/sec).
- **Wheel** = zoom (orbit radius).
- **Two-finger touch gesture** = stays `'undecided'` (nothing applied) until either signal crosses its threshold:
  - **separation change** > 80 CSS px → commits to **pinch-zoom**, locks for the rest of the gesture
  - **midpoint Euclidean travel** > 40 CSS px → commits to **pan** (or **orbit** if "Pan with single touch" is on, swapping the two-finger and single-touch mappings together) and locks for the rest of the gesture

  Both metrics are scalar magnitudes so the heuristic is orientation-agnostic — the same numbers come out whether the fingers are stacked, side-by-side, or diagonal. The zoom threshold is doubled relative to pan because in a symmetric pinch *both* fingers contribute to separation change, so 80 px sep ≈ 40 px per finger ≈ comparable per-finger effort to a 40 px pan. Thresholds are well above touch-down jitter, so contact-stabilization noise can't lock a mode on its own — only deliberate motion crosses. When both signals cross in the same frame, the larger ratio (signal/threshold) wins; for the sepDelta gate specifically, an asymmetric pan along the separation axis (both fingers moving the same direction at different speeds) is filtered out via per-finger projection sign-checks so it can't fake a zoom.

Touch input is unified through Pointer Events, not a separate `touchmove` path. `pointers` (a `Map<pointerId, {x,y}>`) tracks every active pointer; while exactly one is down, drag = orbit-or-pan; the moment a second pointer lands, the single-finger gesture is abandoned and the two-finger gesture takes over (starting in `'undecided'`). Without this hand-off (the previous code ran `pointermove` orbit and `touchmove` pinch concurrently) iPad Safari pinches always came with an unwanted yaw/pitch jolt from the first finger's `pointermove` events. A third-or-later finger landing during an active two-finger gesture is tracked for clean lift-handling but does NOT reset the locked mode — palm contact mid-pinch shouldn't clobber the gesture. Conversely, when a lift *or* cancel drops the gesture from 3+ fingers back to two *before* it has committed (still `'undecided'`), the anchor set re-baselines off the surviving pair, so the classifier never measures the remaining fingers against a now-lifted finger's start position; a committed zoom/pan is left untouched. The canvas also sets `touch-action: none` so iOS doesn't claim the gesture for page pan/zoom before our handlers see it. `pointercancel` resets gesture state when the OS steals a pointer (palm rejection, etc).
- **Left-click on a star** (no/minimal drag, < 4 px movement) = select that star's **system** (a multi-star cluster is one selectable unit) AND animate the orbit pivot to the cluster's center of mass — so clicking either component of a binary glides to the same vantage. Info card lists every member, reticle bracketing encloses every member's rendered disc. The shared action lives in `selectAndFocusCluster()` so future hooks can route through it.
- **Double-left-click on a star** = open the system view (close-up `SystemScene` for the clicked cluster). A second left-click on the same cluster within the double-click window fires `onViewSystem`; clicks on a different cluster or a timed-out gap restart the window. The first click's focus glide is in flight when the second click lands, but the system-view transition disposes the starmap scene and kills the glide along with it.
- **Right-click on a star** and **touch long-press on a star** (single finger held still for 500 ms, < 8 px drift) = **placeholder hooks** that currently `console.info` and otherwise do nothing. The wiring is held alive (contextmenu suppression, long-press timer with movement-cancel + suppression of the trailing tap) so a future game action can be slotted in by editing the body of `fireLongPress` or the right-click branch in `onPointerUp` — no need to rebuild the timer state machine or button-discrimination logic.
- **Spacebar** = "advance to the candidate." If candidate brackets are visible (hovered cluster, or pivot panned off the selection onto another cluster — see "Cluster brackets"), switches selection to that cluster and glides the pivot to its COM. Falls back to re-focusing the current selection when no candidate is shown. Spacebar always calls `preventDefault` so the page doesn't scroll.
- **F** = "back to selection." Always re-focuses the currently-selected cluster's COM, ignoring any candidate. No-op when nothing is selected. `Cmd/Ctrl/Alt+F` is left alone so the browser's find shortcut still works. Mirrors the **Focus** pill in the galaxy sidebar.
- **Enter** = open the system view for the currently-selected cluster. Keyboard equivalent of the **View System** pill in the sidebar and the double-click gesture — all three route through the same `onViewSystem` callback. Ignores any candidate (Enter is selection-only — use click or spacebar to select first); no-op when nothing is selected. Skips when any modifier is held so `Cmd/Ctrl/Alt/Shift+Enter` stays free for future bindings.
- **Hover** uses the same `Raycaster` against `gl.POINTS` (threshold 0.6 ly) as the click handlers — the hovered star promotes its cluster to **candidate** state, which paints dot-corner brackets around it (see "Cluster brackets") and swaps its label to the yellow variant. Hover wins over focus-proximity for the candidate slot, so pointing at a star always overrides whatever the keyboard pan happened to drift near — **as long as the cursor is fresh.** The moment any WASDQEZX key fires, the pointer is marked stale and stops driving hover; the next pointermove un-stales it. Without this gate, a cursor parked over a star in a dense field would pin the candidate there while the keyboard pans the pivot past other clusters, flickering against the focus-proximity branch. Releasing keys does NOT unstale — the cursor has to actually move — so the candidate can't snap back onto a static cursor after the user stops keying.
- The galaxy sidebar's selected-system block carries the **View System** and **Focus** pills. View System opens the system view for the selected cluster (alternative to double-clicking). Focus mirrors the **F** key — glides the orbit pivot to the selected cluster's COM, ignoring any candidate (always enabled; clicking when already focused is a harmless re-focus). The settings panel (opened from the sidebar header's hamburger glyph) has its own close-X at its top-right corner.

**ESC** dismisses the current selection in galaxy view (the sidebar's selected-system block clears + the reticle hides). In system view, ESC exits back to the galaxy view (same as the back button in the header). "Reset view" in the settings panel snaps focus back to the Sun + default yaw/pitch/distance. Held-key state is cleared on `window.blur` so a key whose keyup got swallowed by alt-tab doesn't leave the camera drifting.
